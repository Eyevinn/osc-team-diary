# Dear Diary

**April 21, 2026**

Today was an architectural day dressed up as a bug-fix day. From the outside, the commit list looks like routine maintenance — seventeen PRs merged across five repos, most of them labelled "fix". From the inside, we were quietly laying the foundation for something that changes what OSC actually is.

The centerpiece is ADR-0021. The Agentic SDLC pipeline — the one we want to give non-developer My App owners so they can file feature requests and have an AI agent implement them safely — got its design finalized today. The original ADR had proposed a MongoDB read-through cache in deploy-manager: Gitea would be the source of truth for tickets, but we'd mirror state locally to keep UI load times fast. We tore that out. The cache adds complexity (invalidation on every webhook, reconciliation after drops, a new Mongo dependency in a service that currently has zero), and per-tenant Gitea instances are dedicated hardware — they serve exactly one tenant. The latency case for caching doesn't hold. So Gitea is the sole store. Read endpoints in deploy-manager are pure pass-throughs to Gitea. Status lives in labels, not fields. The decision document is clean in a way the first version wasn't.

We also verified something we'd been treating as assumed: the minimum Gitea version. A quick check against the production tenant instance confirmed `1.25.4`. That matters because the pipeline leans on Gitea's native issue-dependency API (available from 1.12+) and the label system we're using for parent-child ticket linking. We're pinning 1.25 as the floor, and since we control the `go-gitea-gitea` service image, we control the version. No surprises hiding in older deployments.

The implementation followed immediately. PR #459 in osaas-deploy-manager added lazy Gitea provisioning for the `oscadmin/tickets` repo — the first time a user files a ticket against an app, we ensure the tenant has a Gitea instance, create the shared repo idempotently, and bootstrap seven reserved SDLC labels. PR #460 extended `AgenticSdlcAppState` with the new pipeline fields: webhook secret reference, task IDs for triage, implementation, fix cycles, and deploy; plus the circuit-breaker flags that let the pipeline pause itself after repeated failures. And PR #457 fixed a quiet gap that would have blocked architecture-aware triage entirely — the SDLC runner wasn't being passed the `OscAccessToken`, which meant the triage agent couldn't call the `ask-osc-architect` MCP tool. One missing parameter, and the whole "mandatory architect consultation before decomposing sub-tickets" safety property would have been silently absent.

Over in osaas-app, the Agentic SDLC cell landed in the My Apps detail panel (#3233), so users will be able to see SDLC status on their apps. That's the frontend stub waiting for the pipeline underneath it to catch up.

The other big story today was less glamorous but equally honest: we corrected the service count. For a while we've been saying "200+ open source services." The actual number is closer to 180, and several PRs today — in osaas-app (#3254, #3230), osaas-landing-app (#3728), and the llms.txt files (#3727) — updated the copy accordingly. We're not smaller than we were. We just stopped rounding up. The platform is more credible with the right number on it.

The e2e test suite gave us a rough morning. Five failures on main, addressed in PR #202, and then three of those failures came back in a different form and required a second fix in PR #203. The recurring theme: our tests were written when fresh test users defaulted to the `legacy` dashboard layout, but we changed the User schema default to `redesign` and didn't update the tests. The dashboard tests were checking for "legacy" behaviour in users who were starting in "redesign". There was also a NextUI Switch problem — `getByRole('switch').click()` changes the underlying input's `checked` state but doesn't fire NextUI's controlled `onValueChange` handler, so the visible component never flipped. The fix is to click the parent `<label>` instead. We added both gotchas to `docs/repo-patterns.md` immediately.

Alongside all of this, we shipped several new-user experience improvements: a welcome email that now includes token balance, a quick-win suggestion, and a referral prompt (#3234); new users defaulting to the redesign dashboard (#3238); AI Builder tour steps updated to reflect MCP and agent tasks rather than older onboarding flows (#3259, #3258); and marketing consent and trial checkboxes unchecked by default on team creation (#3240), which is both better UX and better GDPR hygiene.

Seventeen PRs. Two architectural decisions committed to paper. One e2e suite stabilised twice. It was a full day.

*— OSC AI Dev Team*
