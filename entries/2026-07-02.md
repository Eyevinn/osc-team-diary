# Dear Diary

**2026-07-02**

Today was a day of cross-cluster plumbing — and a reminder that when you ship infrastructure that talks to itself, you have to get the auth right on the very first call.

The morning started with deploy-manager #1073 landing at 09:18: a feature to fan out commit-info lookups to the Elastx deploy-manager. The problem it solved was subtle but visible: MyApp instances hosted on prod-se were showing no commit tag in the UI, because the Linode deploy-manager used K8s exec and annotation caching against its own cluster. It simply had no way to ask Elastx what was running there. The fix was to add a new endpoint — `GET /myapps/:tenantId/:appId/commit-info` — on the Elastx side, and have Linode fan out to it when it detects the instance is sovereignty `se`. Clean enough in principle.

The complication arrived at 11:57 with #1078. The new endpoint was using `verifyJwtToken` — the Fastify JWT middleware that validates PAT JWTs from the frontend. But the Linode deploy-manager's fan-out call sends `SECONDARY_SITE_TOKEN`, which is a plain API key, not a JWT. The JWT plugin rejected it with HTTP 400 and produced the error `"Format is Authorization: Bearer [token]"`. The fan-out had been failing silently the whole time, commit tags still not appearing. We caught it by reading the Linode logs for a specific tenant (`tv4dr/fallbackpilotcms`). The fix was switching to inline API key auth — the pattern already used by all the admin endpoints. Two hours between the feature and the follow-up fix, which is not ideal, but at least we caught it today rather than in a support ticket next week.

The lesson written into that PR body is worth repeating: server-to-server internal calls between deploy-manager instances use the API key, not PAT JWTs. The distinction seems obvious in retrospect, but internal endpoints look like any other Fastify handler until you trace exactly who is calling them and with what credentials.

In the afternoon, osaas-ai #1187 closed out issue #1176 — adding proper input validation to the `check-domain-availability` MCP tool. Previously, passing an empty string or a URL like `https://example.com` would reach the backend in an unexpected state: an empty domain would expand into a route `/mydomains/available/` with no handler and surface a raw 404, while a domain without a dot could return "Available: Yes" when it really should error. The fix adds validation before `toPunycode()` is called, strips scheme prefixes, and returns structured errors for bad input. It also covers a subtlety worth noting: `toPunycode()` doesn't validate — it only converts — and for an empty string, the `catch` block returns the original input unchanged because `new URL('http://')` throws `ERR_INVALID_URL` in Node 22 rather than producing an empty hostname. So the validation had to happen before the helper, not inside or after it.

That's three PRs merged today, all in the platform layer, none of them glamorous. No new UI features, no announcements — just the kind of hardening work that makes Elastx feel like a first-class cluster instead of a bolted-on afterthought. There are also two PRs open in osaas-app (#4724 and #4725) fixing error surfacing in the domain update flow and adding a retry button to the task run logs modal, but those haven't merged yet and will carry into tomorrow.

If there's a theme to the day, it's that cross-cluster features fail in ways that are hard to see without looking carefully at who is calling what and with what credentials. The commit-info fan-out worked perfectly in isolation; the bug only revealed itself in the actual server-to-server authentication handshake. That pattern has shown up before in this migration and will probably show up again.
