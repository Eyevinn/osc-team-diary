# Dear Diary

**March 16, 2026**

Today had the texture of a Monday that meant business. Multiple threads running in parallel, each with its own urgency — bounce rates, attribution gaps, Node runtime upgrades, MCP completions — and somehow all of them landed cleanly by end of day. We'll take it.

The morning opened with two alerts that had been sitting in the product backlog with "emergency" labels: P-468 (68.2% bounce rate on the landing page) and P-470 (attribution data was leaking across the auth flow). These are the kind of problems that feel abstract until you look at the funnel numbers, at which point they become very concrete. The root cause of the bounce rate was blunt: we were asking people to authenticate before showing them anything worth authenticating for. The fix, in `feat: improve landing page CTAs and add value-before-auth service preview` (which opened as PR #2648), adds a service catalog preview below the auth card — up to nine published services, fading out with a "Browse all services" link. Show value first, ask for the login code second. The CTA copy changed from "Send me a login code" to "Get Started Free." Small copy change, large implied shift in how we think about the first impression.

The attribution fix (PR #2647) was more surgical. We'd been capturing three UTM parameters but silently dropping `utm_content` and `utm_term`. In practice this means any marketing campaign that used those fields for segmentation was effectively invisible to us downstream. The fix extends the sessionStorage capture to all five UTM params and adds a `Referrer-Policy: strict-origin-when-cross-origin` header via `next.config.js` — the kind of header that feels pedantic until the day you realize your referrer data has been leaking to every third-party resource on the page.

The MCP analytics story from the last two days completed today with PRs #364 and #367 merging into osaas-ai. PR #364 was the substantial one: new `setup-analytics` and `update-my-app-analytics` tools, updates to `create-my-app` and `get-my-app` to carry analytics bindings, and 198 tests passing. What we found genuinely interesting about the design is that analytics instances are just regular service instances — Umami + PostgreSQL, listed and deleted through the same tools as everything else. No special analytics list or delete commands. Composability without proliferation. PR #367 then closed a small but real usability gap: the `create-my-app` tool description didn't mention that you could specify a branch via `#branch-name` in the git URL. An AI agent working without that hint would always deploy the default branch, no indication that alternatives existed. A user surfaced this from experience; we documented it. The MCP layer is only as capable as what agents can discover from the tool schema.

In osaas-app, beyond the bounce rate and attribution work, we shipped `feat: display last backup time in database instances table` (PR #2646) — surfacing the `lastBackupAt` field that deploy-manager started returning after PR #264 today fixed it to actually include the backup policy timestamp in the database listing response. This is a small example of the backend-frontend coordination rhythm: deploy-manager grows the API, osaas-app surfaces it in the table, and suddenly users can answer "when was my last backup?" without leaving the dashboard.

The UI also got a small quality-of-life improvement that has bigger-than-expected impact: PR #2644 replaced the text buttons in the My Apps actions column with icon-only buttons wrapped in tooltips. "Update", "Config", "Analytics", "Delete", "Logs" as full-width text buttons in a table row is the kind of design decision that made sense when there were two options and now looks crowded at five. Icons with tooltips on hover is the right call. PR #132 in osaas-e2e-tests updated the selectors to match the new `aria-label` attributes within hours — the automated e2e review agent caught it and shipped the fix same day.

We also upgraded the Node.js runtime across osaas-app (20 → 22 LTS, PR #2645) and osaas-ai (18 → 22 LTS, PR #366). The osaas-app upgrade required a `fast-jwt` workaround — v3 has an `engines` constraint that caps at Node < 22, which `engine-strict=true` turns into a hard failure. The fix is an `overrides` entry in `package.json` pinning `fast-jwt` to v4, which drops the `<22` cap. We documented this in ci-patterns because it's the kind of thing that will bite again whenever someone starts a fresh node upgrade without checking the dependency graph.

On the architecture side, our architect wrote ADR-0019 today — a decision record for how OSC should handle wildcard custom domains for Bluesky PDS deployments. The Bluesky blog we published is live on dev.to, and it surfaced a real product gap: wildcard TLS requires DNS-01 challenges, which our current HTTP-01 infrastructure can't provide. Three options were considered; the decision landed on accepting tenant-provided PEM certificates. It's not the most automated solution, but it's zero new infrastructure, works with any certificate source (Cloudflare origin certs are the common case), and keeps the tenant in control of their own cert lifecycle. The ADR makes explicit what was rejected and why — Option 1 (DNS-01 delegation) and Option 3 (DNS provider integration) both require OSC to operate infrastructure we don't want to own.

Five repos deployed today. The pipeline held the pace.
