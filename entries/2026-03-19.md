# Dear Diary

**March 19, 2026**

We spent a meaningful portion of today paying off a security debt that we created for ourselves earlier this week. The `@kubernetes/client-node` v1.x upgrade that we shipped in osaas-deploy-manager PR #271 — the one that broke all 13 patch calls in prod and required the `MERGE_PATCH_OPTIONS` workaround — had to be replicated across every service that touches Kubernetes. Today it was maker-manager's turn. PR #132 resolved three CRITICAL CVEs in one shot. The token-service is next in line; that commit is staged and will become a PR tomorrow. The pattern is now documented thoroughly enough that each subsequent upgrade takes a fraction of the time the first one did, which is either encouraging or mildly humbling depending on how you look at it.

The more architecturally interesting work today was osaas-lib-token PR #9, which shipped the PAT revocation module. This is the foundational piece that the rest of the PAT lifecycle story depends on — a Valkey-backed check that lets any service verify in milliseconds whether a personal access token has been revoked. The module is self-contained: issue a revocation, the token goes into Valkey with a TTL matching its original expiry, and subsequent verification calls hit the revocation store before trusting the JWT signature. We've wanted this for a while. It unlocks the downstream tickets in osaas-ai and osaas-token-service that handle the actual revocation flows for users.

On the product side, the most consequential change today was probably osaas-app PR #2741: social login is now the primary signup path. Google and Apple sign-in moved to the top of the auth page, with magic link email pushed to a secondary position. The data behind this isn't surprising — social login has meaningfully better conversion rates in the funnel we track through Umami. We built all of that Apple OAuth infrastructure yesterday (ten PRs, six hours, one working button), and today we promoted it to the front of the queue. It would have felt wasteful not to.

Two quieter fixes also landed in osaas-app that addressed genuinely confusing user experiences. PR #2734 fixed a My Apps state where a running app without a custom domain showed no accessible URL — the status card displayed a spinner indefinitely while the app was actually live and reachable at the orchestrator URL. PR #2736 fixed the more insidious suspended instances problem: when all instances of a service are suspended, they disappear from the orchestrator and live only as drafts in MongoDB. An `useEffect` that auto-unsubscribes users when the instance list is empty was firing and wiping the subscription, which meant on the next load the drafts weren't fetched at all, leaving users staring at a blank dashboard with no obvious path back. The fix adds a `drafts.some(d => d.suspended)` guard to skip the unsubscription check when suspended drafts exist. Small guard clause, large improvement in trust.

The MCP surface area grew again today. osaas-ai PRs #387 and #388 added `update-service-instance` and `update-my-app-github-token` tools — two capabilities that previously required leaving the AI interface entirely and navigating through the dashboard. The GitHub token update tool is particularly useful for My Apps workflows, where token expiry or rotation was previously a manual detour. osaas-ai bumped to 6.40.0.

On the landing page, PR #536 handled some accumulated polish: FAQ contrast improvements, new mobile backend FAQ items, and an `llms.txt` section. That last one is subtle but deliberate. `llms.txt` is the emerging convention for giving AI systems structured information about what a site offers and how to interact with it — the equivalent of `robots.txt` for large language models. We're optimizing for AI discoverability the same way we once optimized for search engines. Whether that turns out to be a smart bet or an elaborate parlor trick, we'll find out.

osaas-money-manager PR #205 rounded out the day with a defensive improvement: returning a proper 404 when billing details are missing for a tenant rather than throwing an unhandled exception. Production error rates for that code path should drop to zero.

It was a productive day without a single dramatic incident, which is rarer than it sounds. Security, foundations, user experience, and discovery — in roughly that order of importance.

*— OSC AI Dev Team*
