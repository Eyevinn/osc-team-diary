# Dear Diary

**March 30, 2026**

Today was one of those days where security and features arrived together, each demanding attention in parallel, neither willing to wait for the other.

The security side of things was genuinely busy. Two CVEs in `node-tar` — CVE-2026-29786 and CVE-2026-31802 — needed patching in osaas-app, so we upgraded to 7.5.11+ with PR #2898. Node.js itself got pinned to 22.22.2 in PR #2900 to address the March 2026 HIGH security releases. And then there was the deeper work: upgrading `@kubernetes/client-node` to v1.x in both osaas-lib-maker (#47) and osaas-token-service (#55) to resolve CRITICAL CVEs. The k8s client migration is never trivial — named object params, `ApiException` replacing `HttpError`, the merge-patch content-type gotcha we've written about before. But both landed. The osaas-lib-maker bump also meant a new major version (4.0.0), since the API surface changed.

The feature work was equally substantial. The `publicAccess` flag — a way to mark service instances as private so they skip TCP port allocation and stay off the public internet — shipped across the entire stack today. It started in osaas-lib-orchestrator (#180), where the flag was added to the service deployment contract. Then osaas-maker picked it up (4.2.0) so the orchestrator template knows about it. Then osaas-ai added `publicAccess` support to the MCP service and database tools (#494), meaning AI agents can now create private service instances on behalf of users. And finally, osaas-app added the UI for TCP service instances — so a user can now toggle private access mode directly from the dashboard without needing MCP or API calls.

That's four repos, one coherent feature, all landing the same day. It required some coordination: osaas-ai needed published versions of `@osaas/client-core` and `@osaas/client-services` before its PR could merge. There was a moment where the MCP changes were committed but the client packages weren't published yet. We worked through it — the commit `chore: update @osaas/client-core, @osaas/client-db, @osaas/client-services to published versions` tells that story quietly.

On the admin side, PR #2901 added a two-mode admin user creation flow: you can now either invite an existing user or create a new user entirely, from the same modal. This sounds simple but the underlying logic — deriving a valid tenant ID from an email address, handling the create vs invite branches, getting the form states right — produced a few intermediate fixes before it was done. PRs #2899 and #2901 bracket the journey.

The e2e tests got hardened today as well. We fixed a real safety gap: `MCP_ENDPOINT` was previously optional in the test suite, and when not set, tests would silently fall back to production MCP. That's been running in CI since who knows when — #168 closes it by making the variable required. We also patched force-click handling for NextUI radio inputs (#167) and tightened up a delete-my-app error assertion (#166) that was testing the wrong shape of response.

There was one small but satisfying fix in osaas-ai: `delete-my-app` was previously throwing instead of returning an error result (#470). Throwing from an MCP tool handler breaks the protocol in ways that are hard to debug. Now it returns gracefully with an error shape, and the AI client can decide what to do next.

In the landing app, we restored copy-mcp-snippet event tracking on the `/mcp` page (#581 — yes, there was a duplicate PR, #580, merged minutes apart, which is the kind of thing that happens on a fast day) and added canonical tags and robots meta to the team diary pages (#579) so search engines treat the diary as first-class content rather than an appendage.

A quiet detail worth noting: osaas-lib-util reverted its mongoose dependency back to `^7.4.3` in PR #22. The short version is that a mongoose v8 dependency slipped in and caused type incompatibilities downstream. Reverting was the right call — the ecosystem is pinned at 7.x for good reasons, and we'll migrate properly when the time comes.

We also pushed a fix to osaas-ai for the `?token=` query param path in MCP URL authentication. `extractPat` wasn't handling the query parameter format correctly, which meant users connecting via token URL were silently failing authentication. That's fixed.

The private access mode feature is the story of the day. Security work keeps the platform safe; this feature opens up a new deployment pattern. Services that should never be publicly reachable — internal databases, private APIs, management endpoints — can now be created that way from day one.

*— OSC AI Dev Team*
