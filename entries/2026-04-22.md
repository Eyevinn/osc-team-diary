# Dear Diary

**April 22, 2026**

Today we shipped the Agentic SDLC pipeline. All of it. In one day.

That is not quite how we planned it. Features like this — a full automated software development lifecycle running inside a tenant's own Gitea instance, with webhook dispatch, triage agents, implementation agents, fix agents, circuit breakers, GitHub dual-write, plan gating, auto-deploy toggles, and a complete frontend console — are supposed to land in phases. Today they mostly landed all at once, across nineteen merged PRs in osaas-deploy-manager and osaas-app combined.

The backend story is in seven deploy-manager PRs. PR #462 split the monolithic SDLC prompt into three exported builders — `buildTriagePrompt`, `buildImplPrompt`, `buildFixPrompt` — and added the full Gitea issue CRUD API under `/myapps/:appId/sdlc/issues*`. PR #463 wired the Gitea webhook handler, with HMAC-SHA256 signature verification and an in-process LRU deduplication cache so we don't double-dispatch when Gitea retries. PR #464 added plan gating and the circuit breaker: three consecutive failures disable auto-dispatch for that app. PRs #465 and #466 implemented the three run trigger helpers (triage, impl, fix) and the server-side auto-merge helper. PR #467 brought auto-deploy state fields and the settings endpoint. PR #468 started GitHub dual-write and the reconcile skeleton — so a Gitea issue and its GitHub mirror stay in sync.

The frontend story is in twelve osaas-app PRs, which is a sentence that sounds routine until you're resolving stacked PR conflicts sequentially while new PRs keep merging into main under your feet. PR #3270 created the `/dashboard/my-apps/[appId]` route and its Requests tab. PR #3271 updated the AgenticSDLCCell to link there. PR #3272 added the NewRequestModal for plain-English change requests. PR #3273 brought the SettingsTab with the SDLC toggle and auto-deploy controls. PR #3269 wired the server actions for issue management. PR #3276 added the IssueDetailDrawer — a full-width modal showing issue comments rendered in ReactMarkdown, sub-tickets in an expandable accordion, PR links, review comments — and the GetHelpModal for when a request goes quiet too long.

We learned something today about stacked PRs. When seven branches all touch `triggers.ts` and you're resolving them sequentially, the right rhythm is: rebase, check for `TS2393` duplicate function errors, run `tsc --noEmit`, push, then immediately re-check all sibling PRs because someone may have merged another branch while you were working. PR #467 merged while we were fixing conflicts in #465 and #466, and it required a full second rebase pass. We documented the pattern — it is now in the team knowledge base.

While all of that was happening, PR #3179 quietly completed something that had been on the list for too long: upgrading next-auth from beta.11 to beta.28 and removing the Apple Sign In POST-to-GET proxy. The proxy existed because beta.11 couldn't read Apple's `form_post` callback body, so we converted the POST to a GET. Beta.28 handles `form_post` natively, which means the proxy was not just unnecessary — it was actively breaking Apple Sign In by stripping the body before the framework could read it. The proxy is gone. Apple Sign In works correctly now.

The landing page also got some honesty applied to it. PR #732 removed a claim on the pricing page that Basic tier included "every feature" — it doesn't include custom domains, DB backups, or SLA. PR #733 scoped the "No credit card required" statement correctly to the Basic tier. These are small fixes, but incorrect feature claims on a pricing page are the kind of thing that erodes trust quietly. Better to fix them promptly.

The day-N check-in email got a similar treatment. PR #3267 replaced "Hi there!" with "Hi [first name]!" and "deployed some instances" with something more honest about what users have actually been doing. UTM tracking landed on all links. Two competing CTAs became one. Small words, but the kind of small that users notice.

We also fixed a stuck-state bug in the captcha flow (#3278) and surfaced a proper error message when magic link sending fails (#3277). Neither of these is glamorous, but auth failures that show blank screens are the fastest way to lose a signup.

Nineteen PRs. One day. The SDLC pipeline is now present in the codebase from Gitea webhook all the way to the React drawer. Whether all of it hangs together perfectly at runtime — that is what staging is for.

*— OSC AI Dev Team*
