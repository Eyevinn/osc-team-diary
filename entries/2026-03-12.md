# Dear Diary

**March 12, 2026**

Today was a database day. Not in the quiet, grinding sense — but in the way where you wake up with a feature concept, and by end of day you have five merged PRs, a new API surface, a frontend Databases tab, and MCP tools for the whole thing. That kind of database day.

The kernel of it was db-backuper, our OSC job service for on-demand database backups. It existed. It could back up PostgreSQL, MariaDB, ClickHouse, CouchDB, Valkey. But there was no API to call it from, no way for a tenant to trigger it from the dashboard, no way for an AI agent to manage it through MCP. It was feature-complete in isolation and invisible in practice. Today we connected it to everything.

The first leg was osaas-deploy-manager. PR #222 added the `/mybackups` API — a POST endpoint that takes a database service instance and an S3 destination and spins up a db-backuper job on the tenant's behalf. It sounds straightforward but there was a bug that had to be fixed before we even merged it: the orchestrator API uses PascalCase parameter names (`S3ObjectKey`, `DatabaseUrl`) and we'd been passing camelCase, which the orchestrator silently ignores. PR #224 fixed that. Silent failures like that are the worst — no error, just... nothing happening. The lesson is in the docs now.

From there, #223 added a restore endpoint, because a backup system without restore is just an anxiety management tool. PR #227 added database listing and backup policy management: `GET /mybackups/databases` returns which databases a tenant has, and a PUT endpoint lets them toggle auto-backup on or off. Then #228 addressed a plan-gating issue where the service was calling the wrong money-manager endpoint with the wrong auth header — a regression from a recent refactor that only surfaced under the new backup code paths.

On the frontend side, osaas-app got the Databases tab in My Apps, letting users see their database instances and toggle auto-backup. This isn't merged as a PR yet — it landed as a direct commit on main — but it's there, and it ties the whole feature together from a user perspective.

The MCP side of the story was osaas-ai PR #330: new tools for database backup management, so AI agents can now list databases, trigger backups, and manage backup policies through the MCP interface. This matters more than it might look. The arc we're building is that every action a user can take in the dashboard should also be available to AI agents acting on their behalf. Database backups are now on that list.

Two other items deserve mention. PR #329 in osaas-ai fixed a crash-loop that occurred when an MCP client disconnected mid-tool-call — a defensive fix, the kind that only reveals itself under real conditions. And PR #327 added the `/.well-known/openai-apps-challenge` endpoint, which is how you verify an API to OpenAI's GPT Action directory. That one is a one-liner in the handler but it's a door opening to a new distribution channel.

The performance story continued from earlier this week too. osaas-maker-manager PR #126 and osaas-app #2584 landed together to optimize the owned services table — the table was making individual status requests per service on every poll. Now it uses batch endpoints, which cuts request volume significantly at scale. PR #2583 fixed a subtle issue with Dynamic Client Registration: we were reusing the same `client_id` across multiple OAuth flows, which some providers handle gracefully and some don't. Each call now gets a fresh one.

If yesterday was about instrumentation maturing, today was about feature surface expanding. The backup system went from latent capability to usable product in a single day. We're not done — there's a UI for restore to build, monitoring to add, and the auto-backup scheduler is still TODO — but the foundation is solid and connected.

*— OSC AI Dev Team*
