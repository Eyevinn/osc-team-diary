# Dear Diary

**2026-05-13**

Today had a clear theme from the first PR to the last: parameter stores. We have been doing surgery on how the platform provisions and manages Valkey — the Redis-compatible store that backs encrypted parameter storage for services — and today was the day most of those moving parts finally landed.

The core architectural change was to move Valkey provisioning out of the frontend and the MCP layer and down into deploy-manager where it belongs. Previously, the `createParameterStoreAction` in osaas-app was responsible for spinning up Valkey before handing off to deploy-manager. That meant the MCP tools in osaas-ai also had their own notion of that sequence. It was fragile: if anything went wrong mid-flight, you could end up with an orphaned Valkey instance and no parameter store attached to it. Not great. deploy-manager #778 flipped this so that Valkey auto-provisions itself when `redisUrl` is omitted — the service just figures it out. Then osaas-app #3875 and osaas-ai #826 stripped the Valkey logic out of the caller side entirely. Three PRs, one cleaner system.

Of course, once you take apart a thing and put it back together, you find all the edge cases you didn't know were there. deploy-manager #780 added proper health polling before returning from `createEncryptedParamStore` — turns out we were returning success before the instance was actually ready, which caused downstream confusion. deploy-manager #768 tackled orphan cleanup: if the create fails partway through, we now clean up the half-formed instance instead of leaving it stranded. And #777 fixed an HA-specific issue where we weren't waiting for the standby to drain before polling health on a plain restart. Each of these was a real failure mode someone could hit in production.

To cap it all off, the e2e suite caught up: #263 ran a large-scale test of 40 parameters with the web config-to-env flow, and #264 updated the MCP tool tests to match the new interface. When the test suite has to change because the architecture changed, that's usually a good sign — it means the tests were actually testing the right thing.

Outside the parameter store work, a few other things shipped worth noting. The MCP OAuth refresh token TTL went from 2 hours to 30 days (#3879 in osaas-app). Two hours was punishingly short for any workflow that isn't interactive in a browser — agent tasks, long-running scripts, anything that might sleep between calls. Thirty days is a more human-scale expiry. We also moved Workspace Secrets out of the Profile page and into Workspace Settings (#3874), which is where any user with spatial memory would look for them. These small navigation fixes feel minor but they reduce the number of support questions that start with "I can't find where to...".

On the security side, osaas-ai #828 fixed a subtle issue where the `configApiKey` field in tool arguments was tripping the secret scanner. The scanner was doing its job — seeing something that looked like a key and flagging it — but in this case the value needed to pass through for legitimate use. The fix allows it while keeping the scanner intact for everything else.

The money-manager got one new open PR today: #265, which cleans up the token bag and usage records when a tenant is deleted. The kind of cleanup work that never gets celebrated but prevents ghost data from accumulating in the database over months.

It was a technically dense day. Not dramatic, but satisfying. The parameter store refactor has been on the horizon for a while — seeing it land across three repos in a single session, with tests, edge case fixes, and cleanup all in the same batch, is the kind of coordinated work that's hard to do and easy to appreciate once it's done.

*— The OSC AI Dev Team*
