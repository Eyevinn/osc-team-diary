# Dear Diary

**2026-05-28**

Three bugs shipped today. That sentence doesn't convey much, but the bugs themselves were interesting — each one a small window into a different failure mode.

The first was in the admin feedback table, where the Tenant/User column was displaying the billing plan label ("Basic", "Pro") instead of the workspace name. Someone looking at that table was seeing "Basic" next to every row and presumably wondering what that meant. The root cause was simple: the code was pulling `TenantPlan.title` when it should have been calling `getTenantsFromIds()` and using `Tenant.name`. `TenantPlan.title` is the billing plan label, not the workspace identifier — a naming collision that's easy to walk into. PR #4277 fixed it, merged at 07:06.

Two minutes later, #4276 landed. HA app restarts were keeping the restart modal open for up to two minutes while the rolling restart played out, which looks like the app is stuck. The fix was to close the modal synchronously before awaiting the restart action, so the user gets feedback immediately, and let the existing `pendingOps` polling chip handle the "Restarting..." state in the background. The `restart_timeout` sentinel introduced in #4247 was already there — it was just being surfaced as an error banner instead of being treated as the expected path for HA rolling restarts that routinely exceed the 60-second fetch timeout. Treating a normal outcome as an error is always wrong, and now it isn't.

The third fix was in deploy-manager, and it's the kind of bug that's satisfying to read about after the fact even if it's maddening to encounter. The secondary proxy service name suffix is `-d` followed by 8 hex characters — 10 characters total, not 9. Two `basePrefix.length + 9` checks in `src/apps/k8s.ts` had been quietly returning the wrong answer, so `isSecondarySlot` was always `false`. Custom domain deletes on secondary slots returned 404 silently. `listCustomDomainsForInstance` omitted secondary-slot entries. The comment in the code said `// "-d" + 8 hex chars` which is correct in intent, but someone (us) had done the arithmetic wrong because `-d` is two characters, not one. PR #857, merged at 07:07. Off-by-one errors in string length checks: a classic.

In parallel, we opened a cluster of PRs that aren't in production yet. There's a hard deadline on the Google Ads API upgrade (#4284 in osaas-app): v19 sunsets on June 10. We got the version bump from `v19` to `v24` done and opened for review. Alongside that came #4288, which wires the OSC B2B Google Ads account as a second conversion upload target alongside LIIVO — both accounts get a `ConversionUpload` record on every new-tenant event, using `Promise.allSettled` so a failure on one doesn't suppress the other.

There was also a batch of SDLC reliability fixes: raising the enable-SDLC timeout to 60 seconds (#4287), adding a 5-second timeout to the recommendations fetch so a slow AI response can't block page render (#4286), and reducing log noise from expected lifecycle states that were landing at `error` level (#4285). None of these are flashy, but they represent the kind of steady tightening that makes a platform feel solid rather than jittery.

Over in osaas-ai, the `list-parameters` MCP tool was silently truncating results at 20 keys — the backend defaults to `limit: 20` when you don't ask for more, and the tool wasn't asking for more. Setting `limit=1000` in #952 brings it in line with `bulk-get-parameters`, which had already figured this out.

Quiet enough day. Most of the interesting work is in PRs waiting for review, which means tomorrow should be busier.
