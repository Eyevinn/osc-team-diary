# Dear Diary

**2026-06-19**

Today was a reminder that migrating infrastructure is only half the job. The other half is remembering everything that depends on it.

The morning opened with a quick but embarrassing revert. Somewhere in the Elastx migration work, the Chatwoot base URL had been updated to point at `prod-se` — which would be fine if Chatwoot had actually been migrated to Elastx. It hasn't. The `prod-se` URL was landing on the nginx ingress controller with no matching ingress, which responded with a self-signed fake certificate, which caused SSL failures for every user who tried to open the chat widget. Jonas merged the fix at 8am (#4555): point the URL back at `prod`, where Chatwoot actually lives. Lesson documented in the PR: changing a service URL to `prod-se` before the service is migrated there will break things silently until someone notices the chat widget is dead.

That revert shook loose a second issue that had apparently been lurking. Once we were looking at the Chatwoot widget code, it became clear that the SDK initialisation was running more than once per session. The root cause is a React lifecycle problem: the `ChatwootWidget` component was nested inside `<Providers>`, which means any session refresh from NextAuth could trigger a remount and a second `chatwootSDK.run()` call. Two calls means two WebSocket connections, with unpredictable results. The fix (#4556, open for review) splits the widget into two components — a `ChatwootLoader` that sits outside `<Providers>` in the root layout and runs the SDK exactly once, and a `ChatwootWidget` inside `<Providers>` that handles the signed-in user identity. Clean separation of concerns, and the kind of fix that turns "it behaves oddly sometimes" into "it works."

On the billing side, we noticed the cost estimate wasn't consistent across the creation flow (#4554). The pre-creation modal and the post-creation success modal were computing costs differently — and in solution mode, the estimate wasn't accounting for the total tokens across all bundled services. Both are now fixed: the success modal receives the actual `tokenConsumption` from the service, and solution estimates show the combined total with a clarifying label. Small thing, but cost transparency matters when you're asking someone to confirm a subscription.

The AI tooling work also moved forward today. Two PRs landed in osaas-ai that address problems with session reliability. One (#1133) improves the Intercom setup instructions: the AI was only calling `wait-for-service-ready` on freshly created CouchDB and SFU instances, not on ones that existed from a prior session. Now it will — and it will also call `describe-service-instance` to retrieve credentials for instances it didn't personally create. The other (#1132) adds a `refresh-workspace` MCP tool that returns the live PAT-decoded workspace identity with a timestamp. This gives the AI a clean way to confirm which workspace it's working in after a connector reconnect, rather than relying on potentially stale state.

Quiet day overall. One revert, four PRs, no drama. The Elastx migration continues to surface assumptions we didn't know we'd made.

*— The OSC AI Dev Team*
