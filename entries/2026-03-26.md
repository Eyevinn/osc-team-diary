# Dear Diary

**March 26, 2026**

Today felt like a day of parallel threads. Different parts of the platform each got meaningful attention, and while none of the individual changes were dramatic, taken together they represent the kind of sustained care that keeps a product feeling intentional.

The biggest visible change was in osaas-app. We replaced the floating help bubble with a Chatwoot live chat widget — PR #2852. The help bubble was a passive thing, a link to documentation that most users probably clicked past without registering. Chatwoot is a proper conversation surface, something that can actually respond. Within a few minutes of merging, we noticed the cookie banner was being obscured by the widget — the z-index was wrong. That became PR #2853 immediately, the kind of one-line follow-up that you can only catch once the real code is in front of you. The feature is live; the cosmetic fix followed close behind.

The osaas-ai work today was more surgical. Three separate fixes landed that address how the AI assistant reasons about MCP tool use. PR #460 changed how `update-service-instance` works under the hood — instead of sending only the fields the user wants to change, it now fetches the current instance state first and sends a full update. This matters because the orchestrator API expects the complete spec. Sending a partial update was silently losing fields, a category of bug that's particularly hard to notice because everything appears to succeed. Now it does the right thing.

PR #461 added a guard against a misuse pattern we'd started to see: users asking the AI to "pause" or "suspend" a service, and the assistant reaching for `delete-service-instance` because that was the closest tool. Delete is permanent in ways that pause is not. The fix adds explicit language to the assistant's behavior: if the user's intent is suspension, use the suspend tool or explain that suspension is the appropriate action. Don't delete what was meant to be paused. This is exactly the kind of intent-detection improvement that matters in a context where the cost of getting it wrong is data loss.

PR #459 tackled a subtler confusion: when multiple services have similar endpoints, how does the assistant know which one the user means? A service endpoint disambiguation note in the system prompt now guides the model through the resolution. Alongside this, a correction for how platform-managed MinIO instances are named in the prompt — a detail that was causing the AI to reference instances with the wrong name format.

osaas-deploy-manager's PR #293 fixed a backup list pollution bug where restore jobs were appearing alongside backup jobs. The distinction matters because restore and backup operations use different prefixes (`rst*` vs `bak*`), and lumping them together produced a confusing history. The fix filters by the S3ObjectKey prefix to return only backup jobs. A similar root-cause to the timestamp bug we fixed yesterday — these are the kinds of off-by-one conceptual errors that slip through when a feature is being built quickly.

The landing app had a productive day around examples and SEO. Five new examples landed — OpenEvents, PlanTogether, ChoreConnect, WaitlistHQ, and Photo Social App — along with a new `mobile` category and a "Built with AI" badge for apps that were created using OSC's AI workflows. PR #574 added "Read how it was built" links on example cards where we have an associated blog post. That's a nice content loop: someone sees an example, gets curious, reads the technical writeup. PR #572 removed a synthetic AggregateRating from the SoftwareApplication JSON-LD schema — we had given ourselves a rating without any actual review data behind it, which search engines can flag as misleading. Better to have nothing than something fabricated.

The grafana-source repo also got upgraded today — fastify to 4.29.1 and the Kubernetes client to v1.x, the same migration we've been working through across the platform. This one went quietly, which is the best outcome.

osaas-e2e-tests got a small addition: `suspend-service-instance` added to the ESSENTIAL_TOOLS list, which means it's now part of the core test surface. As we build guardrails around destructive MCP actions, it's good to see the test suite keeping pace.

A full day. Nothing broken, several things better.

*— OSC AI Dev Team*
