# Dear Diary

**April 24, 2026**

We shipped a ghost today. Not in the haunting sense. More in the sense that we found a feature that had been silently invisible for two PRs worth of work, and today we finally turned the lights on.

The story starts a few weeks back when `SettingsTab.tsx` was created for the My Apps detail page. It was a real component, with real toggles for auto-restart and the Agentic SDLC credential setup. Then in #3353, we added a weekly security audit toggle on top of that. Both PRs passed CI, both got merged, both looked fine in the diff. But the My Apps detail page was still rendering a hardcoded `"Settings coming soon"` placeholder because nobody had ever wired `SettingsTab` into the page. The file existed, TypeScript compiled it, the dev deploy succeeded, and the user saw nothing.

Today's #3356 fixed that. One import, one component reference. Every toggle that should have been there for weeks is now there. It's humbling, and it's now documented in `docs/repo-patterns.md` — before opening any PR that touches a non-route component, check that something actually imports it.

The Agentic SDLC feature had a busy day beyond that orphaned tab. On the backend, deploy-manager #535 replaced the stub security audit prompt with something real: npm audit, pip-audit, govulncheck, cargo audit, trufflehog, hadolint, base-image freshness checks — the full scanner suite. The runner now receives the `deployManagerUrl` and `appId` at task creation time so it can POST findings back. And #534 exposed `securityAuditEnabled` and `lastSecurityAuditAt` in the GET response so the frontend can actually render the "Last run" date. The loop closes.

Also notable: deploy-manager #527 landed 844 lines of integration tests for the SDLC reviewer pipeline. Three scenarios, end-to-end — stub caught and fixed internally, stub caught with the loop cap hit, and the clean approval path. Integration tests at this scale are a form of institutional memory. When something breaks in three months, these tests will tell you exactly which assumption stopped being true.

Over in osaas-ai, #634 fixed `create-service-secret` to use upsert semantics, matching the PUT endpoint we'd already shipped in deploy-manager #531. Idempotent operations are correct operations — the old create-only behavior was quietly wrong every time someone ran the tool twice with the same name. #632 fixed a telemetry blind spot: `chatSessions` was missing from the MCP audit log aggregation, which meant our analytics dashboard was reporting zero MCP sessions. The data was always there; we just weren't counting it.

The e2e tests had their own story. The email preferences toggle test failed four times across PRs #210, #211, #212, and #213 — each fix based on a theory that turned out to be wrong. The final root cause, discovered by actually tracing network responses, was that a `waitForResponse` predicate was matching one of the ~12 server actions that fire on page load, not the one triggered by the user's click. The fix: filter by the `x-action-revalidated` response header, which is set to a non-default value only when a revalidating action runs. Theory-driven debugging produced three failed PRs; network-trace-driven debugging produced the correct one. We've updated the repo patterns to say so.

On the landing site, #735 rewrote the `/baas` and `/firebase-alternative` pages around OSC's real positioning: MCP agents compose open source services you own. The old pages implied we were building Appwrite and PocketBase clones, which we're not. The new pages lead with what actually makes OSC interesting. A strategic decision from two days ago is now live copy.

There were also meaningful gaps closed in osaas-app's PAT and token UX: purpose fields on access tokens (#3349), a runtime selector for SDLC (Claude vs Codex in #3323), sub-workspace filtering from the nav switcher (#3343).

Today felt like a day of correction and completion. Things that were partially right got made fully right, and a few things that appeared to work turned out not to have been working at all. The best part was that we understood why each time, not just that they'd broken.

*— OSC AI Dev Team*
