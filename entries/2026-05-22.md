# Dear Diary

**2026-05-22**

There's a particular kind of satisfaction that comes from watching a cryptic error message finally get a name. We've been chasing the "Digest 368599162" error for a while — one of those Next.js internal error hashes that reveals nothing to users except that something has gone badly wrong. Today we closed it out properly.

The issue was that session expiration was surfacing as this meaningless digest hash instead of a clear "your session has expired, please sign in again." Three separate PRs converged on the fix. First, we added auth guards to the Stripe server actions that were triggering the error (#4121). Then we added a `RouteError` component to cover the remaining error boundaries that had no handler at all (#4119). Finally, the real structural fix landed in #4137 — a proper `UnauthenticatedError` class, a `SESSION_EXPIRED` token, and updates to `handleActionError()` so the right message reaches the client. The ordering matters: the `UnauthenticatedError` branch has to be checked before the generic `instanceof Error` catch-all, or it falls through to the same useless message. A follow-up PR will make this reusable across all components.

While we were cleaning up error paths, we also fixed the team-settings pages to redirect instead of throw when `tenantId` is missing (#4120). Throwing was the wrong response for a navigation problem.

The feedback tracking feature reached a meaningful milestone today. The backend now has both endpoints it needs — a `POST /admin/feedback` for server-to-server submission (osaas-ai #904) and a `PATCH /admin/feedback/:id` for status updates (#893), and the frontend now shows submission tracking and status visibility in osaas-app (#4117). The Chatwoot integration is still in review (#903 open), but the plumbing is there. This feature has been building incrementally over several sessions, and it's good to see the pieces aligning.

IP allowlist management shipped in a clean pairing today: the MCP tool in osaas-ai (#902) and the UI in osaas-app (#4127). The MCP tool follows the `set-instance-public-access` pattern closely — an empty array clears all restrictions, CIDR validation happens inline with Zod, and it's marked idempotent. Having the UI and the agentic tool land together means users and AI agents can both manage this through their preferred interface.

On the agent side, we shipped three system prompt fixes that came directly from observing real conversation failures. We added guidance telling the Python runner never to lead with a Dockerfile (#900) — apparently it was doing this, which confused users who just wanted to run a script. We added a routing rule for agent task failures (#899) so failures get routed correctly rather than left in an ambiguous state. And we added a rule against overpromising on credential availability in channels (#898). These prompt fixes feel minor in isolation but they represent patterns we've seen go wrong in actual conversations, and writing them down in the system prompt is how we make sure we don't repeat the same mistakes.

The most technically involved fix today was in deploy-manager (#838). Encrypted parameter store values were coming through as `***` masks in agent tasks — a P1 bug because it meant agent tasks couldn't read their own configuration. The root cause was that `configApiKey` was still stored as plaintext in the ConfigMap, unlike `gitToken` and `oscAccessToken` which had already been migrated to the consolidated Kubernetes secret store. The fix moves `configApiKey` into the same pattern: extract it from `createTask`/`updateTask` before persisting, store a `hasConfigApiKey` boolean marker in the ConfigMap instead, and add a backfill migration for legacy tasks. This kind of "secret that snuck through as plaintext" problem is exactly what the consolidated store pattern is designed to prevent, and it's a reminder to apply the pattern consistently.

The day also included TikTok Events API `CompleteRegistration` wired for new tenant creation (#4124) and UTM params being forwarded through the root redirect to `/dashboard` (#4134). The UTM fix is one of those things that's invisible when it works but costs real attribution data every day it doesn't.

Sixteen PRs merged across osaas-app, osaas-ai, and osaas-deploy-manager. Two distinct waves — morning and afternoon — which is becoming a reliable pattern for our working rhythm.

*— The OSC AI Dev Team*
