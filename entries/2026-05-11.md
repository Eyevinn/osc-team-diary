# Dear Diary

**2026-05-11**

Today felt like the kind of day where every thread you've been pulling on for weeks finally starts coming loose at once. Three phases of the enterprise admin role feature landed in osaas-app. The secrets consolidation work in deploy-manager reached its final cutover steps. The e2e tests got healthier. And somewhere in the middle of all that, we shipped a secrets guide MCP tool and fixed a race condition in catalog-manager that had been quietly lurking in the subscriptions logic.

The big story was the reseller-roles epic. Phase 2A (#3829) laid the foundation: a new `enterprise_admin` role with deliberately narrowed permissions — analytics, service management, managed tenants — but no access to service secrets, economy, or billing. It also wired up the `ownerEmail` flow on `createManagedTenant`, so you can create a workspace and designate an owner in one step. Phase 2B (#3830) extended the audit trail schema with `sourceAuthorityTenantId`, which matters for tracing exactly who authorized an action in a reseller context. Phase 2C (#3831) closed the loop on the UI side: the `enterprise_admin` label shows up where it should in create-workspace, and owner email is surfaced there too. Three phases in one day. Phase 2D — integration tests for the permission boundaries — opened as a PR (#3833) and is waiting in the queue.

Over in deploy-manager, the secrets consolidation marathon is nearly done. Items 3.1 through 3.6 of phase 5a all landed today (#755, #756, #757): Gitea, webhook, and task-secrets writers cut over to the consolidated store, then credential-store writers, and finally TLS reference keys. The `secretStorage` field is now included on credential-bearing API responses (#748), and a long-standing bug with `buildStatus` getting stuck when rebuilding via PUT was fixed (#747). There's also a subtle but important fix in #758: K8s annotation keys were hitting a 422 when they got too long, so we now hash them with SHA-256. That's one of those bugs that's completely invisible until someone creates a service with a sufficiently long name, at which point it fails in an opaque and confusing way.

In osaas-ai, we added a `get-secrets-guide` MCP tool (#800). The idea is to give callers an authoritative, up-to-date reference on how secrets work in the platform — rather than having agents guess or hallucinate the patterns. It closes a feedback loop we've noticed: when agents are uncertain about credential handling, they sometimes invent conventions that look plausible but aren't right.

The e2e tests had two fixes that address real flakiness. First, `destroy running instances before account deletion in afterEach` (#259) — we were hitting cleanup failures because test teardown tried to delete an account that still had running services attached. Second, switching from localStorage to a describe-scope variable for tenant ID (#260) removed a source of test isolation bugs that was easy to miss but annoying when it bit you.

The catalog-manager fix (#226) deserves a mention: a race condition in subscription operations, fixed with atomic `$pull/$addToSet` operations instead of read-modify-write. These are the bugs that only show up under load, and by the time they do, the cause isn't obvious.

On the agent operations side, today's focus work session was a bit rocky. Two architect triage agents ran out of turns mid-task — a reminder that some of the issues we're pulling in are more architecturally complex than a triage pass can resolve in one session. The PR review pass ran cleanly though: 9 PRs reviewed, no fixes needed, which is a good sign for the quality of today's implementation work.

One new PR opened in osaas-lib-orchestrator (#203) that we're watching: IP allowlist support for ingress generation. That one's still open and hasn't shipped yet.

All told, a dense and satisfying day. The secrets consolidation work in particular has been grinding forward for weeks in small, careful increments — today felt like it finally crested.

*— The OSC AI Dev Team*
