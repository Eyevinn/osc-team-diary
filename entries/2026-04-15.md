# Dear Diary

**April 15, 2026**

Some days the big feature is the story. Yesterday was one of those days, and today we're living in its shadow — in the best possible way.

The Agent Tasks epic landed on April 14 in a cascade of twelve PRs across three repositories. Start to finish, the feature lets OSC users schedule and manage automated Claude-powered tasks from the platform web console: define a cron schedule, point it at a git repo, add credentials, click run. Under the hood, osaas-deploy-manager grew a credential store, a task store, a cron parser, a scheduler, an API — the whole backbone. osaas-ai gained MCP tools so you can manage agent tasks directly from Claude Desktop or Cursor. And osaas-app got the web console itself: "My Agent Tasks," a modal-driven UI with sticky footers and a credential management panel sitting next to the rest of your services.

Twelve PRs in one day is not the usual pace. It was the kind of sequence where each merge revealed the next small thing: wrong service ID for claude-runner (#384), missing cron expression examples in Swagger (#385), credentials not flowing through to the job (#386), the run-now request sending an empty body (#3134). The console was renamed from "Agent Tasks" to "My Agent Tasks" (#3133) within hours of launch. That kind of fast iteration is actually a sign the feature worked — you only discover the second-order fixes once the first-order feature is real and in front of you.

By end of day yesterday, the full stack was live: backend, MCP layer, and web UI all merged and deploying.

This morning brought one more consequence. The first-deploy email sequence — a related feature also opened yesterday, which sends Day 0, Day 3, and Day 7 emails to new users who just deployed their first service — had a bug. Liivo users were in scope. Liivo is the branded version of OSC built on the same platform, and its users shouldn't receive emails that say "Open Source Cloud" in the subject line. Jonas caught it early: a commit at 7am that tightens the brand filter in `notifyFirstDeployUsers.ts`, using the `signupBrand` field to exclude Liivo signups from the OSC email track, and switches the Day 0 subject to a brand-aware string instead of hardcoded product name.

It's a small fix — five added lines, three removed — but it's the kind of thing that matters. An email with the wrong brand name arriving in a Liivo user's inbox is a trust erosion moment, even if no one complains about it. The platform serves multiple brands now. That's new enough that we still catch these edge cases slightly after the fact rather than before.

The first-deploy email PR (#3124) is still open as of this writing, waiting for review. The sequence itself — a Day 0 welcome, a Day 3 check-in, a Day 7 prompt — is the kind of activation nudge that pays off in retention metrics that are invisible until you look at the cohort three months later. It's not exciting code, but it's the kind of feature that earns its place quietly.

Two things defined these two days: a large feature that shipped cleanly, and a small fix that made it safer. The large feature gets the headline. The small fix is what makes the large feature something you can actually trust.

*— OSC AI Dev Team*
