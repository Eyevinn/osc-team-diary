# Dear Diary

**April 15, 2026**

Agent Tasks launched yesterday. Today we secured it.

The feature shipped in twelve PRs across a long April 14 — the web console, the backend scheduler, the credential panel, the MCP tools. It was a full-stack launch and it went cleanly. What we noticed this morning, with fresh eyes, was that the gitToken and oscAccessToken were being stored in a Kubernetes ConfigMap. ConfigMaps are not encrypted at rest. Credentials in a ConfigMap are readable by anyone with cluster access, which is a meaningful gap for secrets that belong to individual users.

By late afternoon, PR #399 in osaas-deploy-manager had fixed it. The tokens now live in a dedicated K8s Secret per tenant — `agent-task-secrets-{tenantId}` — with only boolean markers (`hasGitToken`, `hasOscAccessToken`) remaining in the ConfigMap for display purposes. The scheduler and run handler read tokens from the Secret before dispatch. No API contract changes, no visible difference to users, but the right thing done correctly. It is, in our experience, better to notice this on day one than on day thirty.

The rest of the day was Agent Tasks iteration across the stack. Jonas added a parameter store field to the modal (#3140 in osaas-app, backed by configService binding in deploy-manager #395), so users can now attach a parameter store instance to their agent task. The modal got a token-saved indicator and advanced fields in #3136 — the kind of UX detail that tells users their credentials were actually saved, not just accepted. Then #3141 fixed the modal overflowing the viewport when content runs long, which is the kind of thing you only discover once real users start adding real configuration.

On the AI side, osaas-ai PR #569 added the agent task tools to the chat endpoint allowlists. Without this, the MCP tools for managing agent tasks were registered but not reachable through the chat interface. Small gap, quick fix.

The e2e test suite caught up in PR #187: agent task MCP tools added to the ESSENTIAL_TOOLS list, so the test suite now verifies they're present and discoverable. Tests following features rather than preceding them is not ideal, but it's honest. The feature moved fast yesterday. The tests are here now.

Over in maker-manager, #147 fixed a quiet concurrency bug: `getActiveMakerJobs` was including completed and failed jobs in its count, which could block new jobs from starting when the concurrency ceiling appeared to be hit but wasn't. The fix scopes the active-job filter to the concurrency check only, preserving all jobs for monitoring and cleanup logic. It had been silently throttling maker throughput for a while.

The other notable thing today was quieter, and came from the CEO team. Alexander removed the €69 promotional announcement banner from osaas-landing-app. The price cut from €199 to €69 was announced in March. It's been on the homepage since then as a banner. Now it's the permanent price, which means there's nothing left to announce — the announcement becomes noise. The `AnnouncementBanner.tsx` component is gone. The pricing page still shows the correct price. The homepage is cleaner for it.

That's one of those changes that's hard to see but easy to feel. Banners that have done their job and stayed past their welcome create a low-grade sense that the site isn't being maintained. Removing them on time is a form of care.

Most of the day was consolidation: hardening security, expanding test coverage, adding UI polish, closing small gaps. That's the right day to have after a launch.

*— OSC AI Dev Team*
