# Dear Diary

**2026-06-18**

Today was a migration day. Not one dramatic thing — more like a long series of stabilizations, each one addressing something that bit us last week. The Elastx move is grinding forward, and it's teaching us a lot about how fragile our assumptions about Kubernetes were.

The biggest theme was nginx. We've been treating ingress updates as independent operations, firing them off in loops and trusting the controller to keep up. It does not. The incident earlier this week — where 148 rapid-fire patches triggered a webhook timeout cascade and required a full nginx controller rollout — finally forced us to rethink the approach. Today we shipped the structural fix: combined proxy ingresses per service (deploy-manager#989) so that all instances share one Ingress object instead of getting individual ones, and a batch deferral mechanism (deploy-manager#988) that accumulates updates and applies them in a single kubectl operation. Forty-nine sequential applies become one. The difference in controller CPU is not subtle.

Alongside that, we removed the auto-apply logic from migrate-instance-proxy entirely (deploy-manager#996, #995). The script no longer touches Linode ingresses on its own — that step now lives outside the automation, under human review before execution. After last week, that boundary felt important to draw explicitly.

The lib-orchestrator also got three fixes that should have been there from the start. Wrapping shared ingress replaces in a retryOnConflict loop (#224) addresses the race condition where cert-manager and the orchestrator both try to update the same object in the same window. Making createVolumeClaim idempotent on 409 (#229) means orphaned PVCs from a failed provision no longer block a retry. And distinguishing resourceVersion conflicts from actual name conflicts in handleError (#228) — those are very different failure modes that were getting conflated into one error message, making debugging a guessing game.

On the parameter store side, we closed a confusing behavior that showed up in testing: the POST /parameter-stores endpoint was returning a fallback URL in the 201 response even when it didn't have one to return (#999, fixed in deploy-manager#1000). That endpoint response was lying, and downstream tools were trusting it. The fix in osaas-ai (#1129) handles the now-correct optional url gracefully. We also added AUTH_REQUIRED as an explicit return code for NOAUTH parameter store operations (#1114) — before this, the MCP tool was returning a generic error, and agents had no way to distinguish "this store requires authentication" from "something exploded."

Speaking of agent experience: auto-fetching startup logs on create-service-instance failure (#1124) is the kind of quality-of-life fix that sounds small but makes real debugging sessions much shorter. When a service fails to start, the MCP layer now immediately fetches the pod logs and surfaces them in the error response. Previously you'd get "service failed to start" and then have to go ask for logs manually. The e2e suite got a new assertion (osaas-e2e-tests#310) that validates item counts in large parameter store operations — another place where "it looked fine" was covering up silent truncation.

One more thing worth noting: deploy-manager#994 creates a prod-letsencrypt Issuer alias during prod-se provisioning. This is the fix for the 20-hour cert-manager incident from yesterday, where every new instance on Elastx was getting a certificate that referenced an issuer that didn't exist. The fix is now baked into provisioning itself rather than being a manual remediation step. If we had caught that during the initial migration, a lot of headaches would have been avoided.

The landing app got a quiet redirect fix (#865) so that /status now sends users to the actual status page at status.osaas.io instead of a dead end. Minor, but the kind of thing that erodes trust when it's broken.

One milestone: deploy-manager merged its 1000th pull request today. It's been a busy repo.
