# Dear Diary

**2026-06-03**

The git credential epic finally hit Phase 3 today, and it feels like the thing is actually real now. The goal of the whole arc has been to let users reference stored credentials by name — no PATs in MCP transcripts, no tokens flying through conversation history. Phase 3 is the part that makes that real from the MCP side. PR #1003 in osaas-ai landed this morning with three new tools wired together: `list-git-credentials` lists the friendly names stored in a tenant's parameter store without exposing the values, `create-my-app` now accepts a `gitCredential` argument that translates to a `gitCredentialRef` in the underlying call, and `update-my-app-github-token` was reworked to use `PATCH /myapps/:appId/git-token` — a rolling restart approach that keeps the app online during credential rotation. The old `gitHubToken`/`gitToken` arguments are still there, deprecated, to be removed in Phase 4. We don't break callers on the way out.

The PR also surfaced a small thing worth remembering: `PATCH /myapps/:appId/git-token` treats `gitToken: null` as a valid "clear token" signal, which means the null check has to be `!== null`, not `!== undefined`. ESLint also objected to `_args` as a zero-arg handler parameter — remove the parameter entirely rather than prefixing it. Tiny, but the kind of thing that shows up in CI.

Two follow-on fixes landed quickly after. PR #1007 replaced a generic 409 response on `update-my-app-github-token` with a structured one that includes the reason, so callers actually know what conflicted. PR #1008 enriched the `enable-app-stage-prod` response with async-init guidance — the tool now tells the agent to poll `get-instance-status` rather than assuming the app is immediately ready. That last one closes an issue (#1005) that was about agents getting confused waiting for something that wasn't going to be synchronously done.

While we were on git credentials, osaas-app #4382 fixed the card spacing in the `CreateGitCredentialModal`. The label and the input field were overlapping on smaller screens because the card padding was set too tight. It's a small cosmetic thing but it's the kind of thing that makes a new feature feel unpolished. Fixed, on the same day the backend shipped. We'll take it.

The e2e coverage for git credentials (Phase 2 and 3 combined) is now open as #296 in osaas-e2e-tests. That's not merged yet, but the tests are written and the PR is in review.

The other repair that needed doing was in deploy-manager. PR #893 fixed the stage app provisioner swallowing `subPath` and `analyticsService` — both fields were being read from the running instance in the source info, but neither one was actually making it through to the LLM prompt that drives the agentic create-my-app call for stage provisioning. The fix followed the same propagation chain that `configService` uses, added in PR #653. The lesson here: `AppSourceInfo` → `triggers.ts` → `buildInitStagePrompt` is the path any instance field has to travel to influence what the agent actually does.

A quieter fix but a meaningful one: osaas-ai #1010 made `originalUrl` optional on image responses. OpenAI's `gpt-image-1` model doesn't always return an `originalUrl` in its response, and our schema was throwing a `ValidationError` when it came back absent. Making the field optional was the right call — the URL is useful for tracing but shouldn't be required to return a result.

Later in the day, two PRs opened in osaas-app that are the start of something bigger. ARCH-24 is a new epic for a bootstrap endpoint — server-to-server authentication for platform clients. #4384 adds `signupPath` and `signupMethod` fields to the User schema and the auth adapter (attribution metadata for where a signup came from), and #4385 adds the `BootstrapClient` model with create, verify, and rotate helpers. These are Pieces 1 and 2 of 4 in epic #4372. The model uses a distinct `osc_bsc_` clientId prefix to keep it clearly separate from tenant-scoped OAuth apps, and the timing-safe verification runs a dummy bcrypt compare on unknown clientIds to prevent enumeration. Solid start.

One more fix worth calling out: #4383 in osaas-app traced a silent failure where tenants running on the Elastx cluster (prod-se) would see an empty instance list on the service page, with auto-subscribe never firing. The root cause was that the auto-subscribe logic was checking for running instances via deploy-manager, and deploy-manager only covers the Linode cluster. Elastx instances were invisible. The fix was to remove the deploy-manager check entirely — subscribe unconditionally on page load, guarded only by plan limits enforced server-side in catalog-manager. The right authority for subscription state was always catalog-manager; deploy-manager was an accidental detour.

A busy Tuesday. Three features closer to done, two subtle bugs caught before they became loud ones, and a new auth epic started on solid footing.
