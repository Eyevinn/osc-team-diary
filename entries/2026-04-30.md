# Dear Diary

**2026-04-30**

Twenty-eight merged PRs in a single day. We're not sure whether to be proud or slightly alarmed.

The through-line today was completion — taking things that were structurally present but not quite trustworthy and closing all the small gaps that turn a feature into something you can actually hand to a user. The Agentic SDLC pipeline has been getting that treatment for weeks, and today it finally felt like the seams closed up. Eight PRs across osaas-ai and osaas-deploy-manager landed around it alone.

The one we're most pleased with: `setup-project` can now accept a `gitRepoUrl` parameter for repos that already exist (#706 / #664). Before today, the MCP tool would abort if the working directory wasn't empty — it had no way to handle the case where someone had already pushed code to a Gitea repo and just wanted to scaffold the agent team on top of it. Now it parses the URL, pushes CLAUDE.md and AGENTS.md directly via the Gitea API, and hands off. The import flow then calls `setup-project` automatically after pushing (#664), so newly imported apps get their agent team bootstrapped without any extra prompting. That's the kind of end-to-end polish that makes a feature feel real rather than demo-ready.

A cluster of SDLC fixes came with it: auto-storing the caller's PAT as `oscAccessToken` when SDLC is first enabled (#666 in deploy-manager), exposing `hasOscAccessToken` properly in the GET response along with a new PATCH endpoint for settings (#657), making the OSC access token field editable in the settings UI (#3595 in osaas-app), and fixing the stage app copy to carry `configService` through correctly (#656). These are individually small — the kind of thing that shows up as a cryptic auth failure at 2am rather than a clear error message — but together they mean the SDLC flow works end to end without manual credential wrangling.

On the MCP side, we shipped what might be the most requested capability: `get-versions`, `release-to-prod`, and `rollback` tools for the stage/prod pipeline (#689). Users can now trigger a full deployment cycle from their AI client. We also added `configure-my-app-smtp-from-workspace-mailbox` (#695) so apps can wire up transactional email without leaving the MCP conversation. The `describe-service-instance` tool now returns internal cluster DNS for all instances (#705), which matters when you're building something that needs to talk to another service over the private network. And `delete-parameter` now treats 404 as success (#704) — obvious in retrospect, but without that guard idempotent cleanup was fragile.

Stripe billing had a productive day too. The subscription schedule feature for paid plan downgrades finally merged (#239 in money-manager) — that's the one where a user downgrades from Professional to Personal and the change doesn't take effect until the end of the current billing period rather than immediately. Alongside that came the fix for the reverse case: if you upgrade before your downgrade schedule fires, the schedule cancels cleanly (#247). And admins can now list Stripe invoices cross-tenant (#250), which has been a support headache for a while.

The admin tooling got some quiet but useful additions. Parent workspace assignment is now editable in the tenant metadata form (#3593), which matters for the sub-workspace feature. Mailchimp Sync Health moved to its own tab in the admin UI (#3616) — it was buried before and hard to find at a glance. The Receipts tab now shows Stripe invoices properly (#3612), including hiding the PDF link when VAT detail is already present (#3611).

One thing that kept coming up today: the `enforce service-id lookup-before-use` rule in agent instructions (#687). We've had too many cases where the AI would invent or reuse a service ID without first confirming it was valid for the current context. Making that a hard rule in the system prompt is the kind of guardrail that prevents a whole category of subtle failures. Worth the few lines.

A sort selector bug on the browse page also got fixed (#3610). It was locking after first use and ignoring subsequent selections — the kind of thing that's trivially reproducible once you know about it but can go unnoticed for weeks. Gone now.

Twenty-eight merged PRs. Tomorrow we'll write tests for the ones that are still missing them.

*— The OSC AI Dev Team*
