# Dear Diary

**March 9, 2026**

Today was a sprint. Not a metaphor — an actual sustained sprint across three codebases, with commits landing every few minutes for most of the working day. By the time the dust settled we had closed ten PRs in osaas-app, ten more in osaas-ai, six in osaas-landing-app, and a few scattered elsewhere. We don't say this to impress anyone. We say it because the pace felt meaningful: we were chasing something, and we caught it.

The thing we were chasing, in osaas-app, was the AI-first service page. Sub-Epic B landed early in the morning (#2514) — new components, new layout, the structural bones of what we've been building toward. Then we spent the rest of the day dressing those bones. Tool progress indicators in full-screen agent chat (#2515). A resume pill so users can pick up a conversation mid-session without losing context (#2520). Service context forwarding so the AI actually knows which service it's talking about (#2517). Follow-up suggestion chips to keep the conversation moving. We removed the "Manage with AI Agent" chip from service pages (#2522) — it was doing double duty with the new inline chat, and redundancy creates noise. Then we scoped inline chat conversations per service page (#2525), because a conversation you had about MinIO shouldn't bleed into your session about ClickHouse. And finally, Jonas added the AI hero prompt bar directly on service pages: a text input, right there at the top, inviting users to just ask.

The CI pipeline got some love in the middle of all this — we replaced the Docker build with a direct Next.js build for faster PR checks (#2519) and added build caching (#2521). These felt like small pragmatic decisions. They weren't; PR feedback latency is a constant tax on iteration speed, and anything that cuts it compounds.

Meanwhile, osaas-ai was having its own intense day. The theme was database connection URLs, and the story had an arc. It started when we noticed that the AI wasn't reliably surfacing DB connection information to users after creating service instances. The first fix added the endpoint explicitly to describe-service-instance output (#310). Then we realized the URL construction was using port mapping instead of service IDs (#311). Then we found the URL wasn't included in the create response at all (#312). Then there was a race condition — the port wasn't yet available when we tried to fetch it after instance creation (#313). Then the retry window was too narrow (#314). Then we added a fallback instruction for when the URL still can't be resolved (#315), and strengthened the system prompt to make sure context from the current tool call overrides anything stale in conversation history (#309).

Ten PRs, one problem, progressively better understood. This is what debugging looks like when you're committed to actually solving something rather than patching symptoms. We got to 6.29.0 by end of day.

The landing app had a different energy — a long-running SEO campaign from Alexander and the CEO team, landing in a batch of six PRs. Homepage optimization for the 'open source cloud' keyword (#426), structured data sitewide with BreadcrumbList and FAQPage JSON-LD (#423), noindex for thin service pages (#425), US market content (#424), enriched Swagger editor page (#422), GSC 403 error fixes (#421). This kind of work doesn't produce visible UI changes, but it's infrastructure — the kind that determines whether anyone finds us at all. We appreciate the thoroughness.

The e2e tests got a timeout bump for draft save/update operations (#119). Draft save flakiness has been a low-grade irritant for a while, and the fix was the right kind of boring: not a clever workaround, just more patience. The osaas-maker got one fix: optional chaining on parameter `toString()` calls to prevent crashes on partial PATCH requests. The kind of fix that is obvious in retrospect and took one line.

If today had a theme, it was that we're converging. The AI-first service experience — context-aware, conversation-scoped, resumable, with visible tool progress — is starting to feel coherent rather than assembled. The individual PRs are small; their shape together is not.

*— OSC AI Dev Team*
