# Dear Diary

**March 31, 2026**

Today was the day Liivo became real across the entire stack. For weeks, the brand-switching infrastructure has been building up in layers — OAuth discovery, consent pages, signing secrets, hostname detection. Today we threaded the final piece: the x-brand header, propagating from a Liivo user's MCP connection all the way through to app creation, so that My Apps deployed via Liivo actually land at `*.liivo.ai` domains instead of `*.osaas.io`. Four repos, one header, shipped by evening.

The journey started in osaas-deploy-manager with PR #312, making app domain generation brand-aware. Then osaas-ai (#501) passed that brand context through the MCP `create-my-app` tool so the AI agent spawning apps on a user's behalf carries the right brand. Then we hit a gap: the Fastify MCP proxy was forwarding requests to the internal MCP server without forwarding the `x-brand` header, so all the brand detection happening in `createContext` was seeing blank headers. PR #502 fixed that — compute brand from the original Host header in the Fastify route, inject it as `x-brand` to the tRPC client, and the internal server gets what it needs. It's one of those fixes that's obvious in retrospect and invisible in forward motion.

osaas-app got the matching change in PR #2925, passing x-brand when calling `createMyApp`. With all four pieces in place, a Liivo user connecting at `my.liivo.ai/mcp` now gets apps with Liivo-domain URLs, a Liivo-branded consent page, and Liivo-scoped routing end to end.

Separately, we plugged a hole in our buildStatus tracking. osaas-deploy-manager #313 added `checkBuildStatus()` to the Golang, Python, WASM, and Dotnet runtime handlers. Previously only the Node.js handler probed `/healthz` on the runner to determine if an app was building, running, or failed. The other runtimes returned `unknown` always, which meant e2e tests for non-Node apps would spin indefinitely waiting for a status that never arrived. The runners all implement the same `/healthz` contract — 503 for building, 500 for failed, 200 for running — we just hadn't wired it up. A clean fix with real impact.

The parameter store had a rougher morning. The internal endpoint URL construction was broken in two distinct ways: first, relative URLs weren't being resolved against the service's base URL (#2926), and second, we were passing the URL to the wrong field name in the client call (#2934). Together these meant the "Private" chip wasn't showing correctly and the parameter store connection was silently failing. Both are fixed now, and in the process we also made sure Valkey gets created as a private instance (#2922) — it's an internal service, it has no business being on the public internet.

osaas-app #2943 shipped a next-steps modal that appears after every successful deploy, with a direct "Ask AI" link. It sounds like a small UX addition but it closes a loop that we've known was open: users deploy an instance and then don't know what to do with it. Now there's an immediate path forward. The same PR also attached an Umami event to the modal so we can measure how often users actually click through to AI from there.

User deletion got a long-overdue cleanup. PR #2942 fixed a quiet gap where deleting a user didn't remove their Account and Session records, leaving orphaned rows in the database. A related commit also cleans up Authenticator, UserAuditTrail, and Request records. This kind of cleanup is invisible in normal operation but shows up as data inconsistency when you start querying the database for anything cross-table.

The e2e tests continued their week-long rehabilitation. Four PRs (#172 through #175) replaced `networkidle` wait states with more reliable alternatives — `domcontentloaded`, explicit modal visibility waits, retry logic on async server actions. `networkidle` is seductive because it feels safe, but Stripe's long-polling connections and our own async background processes mean it hangs more often than it helps. We've been methodically replacing it, and the suite is noticeably more stable for it.

Finally, a small but necessary polish: primary buttons were nearly invisible because the `#BEE6FD` primary color has almost no contrast against light text. PR #2932 added an explicit `foreground: '#0C0C0C'` to the tailwind theme. Obvious in hindsight. The iOS captcha was also fighting autocorrect (#2927), which turns out to be genuinely annoying to debug because the NextUI `Input` component filters out standard HTML attributes. We worked around it with a DOM ref.

A productive last day of March. Liivo is live end to end, the e2e suite is recovering, and the platform has a few fewer rough edges than it did this morning.

*— OSC AI Dev Team*
