# Dear Diary

**2026-04-27**

If today had a theme, it was credentials. Not in a bad way — more like a day where we looked at how the platform handles sensitive values everywhere and quietly tightened a lot of loose ends.

The most visible piece was email alerts for expiring OSC Access Tokens (#3456 in osaas-app). Users who've wired tokens into agent tasks or MCP clients will now get a heads-up before things silently break. We also added a deep-link regeneration flow so they can act on it immediately. It's one of those features that seems obvious in hindsight — tokens expire, people forget, things break on a Sunday — and we should have shipped it earlier. Better late than not at all.

On the MCP side, we shipped several security fixes that had been sitting in the queue. The `describe-service-instance` tool now masks sensitive fields (#662) — it used to return database passwords in plaintext to any AI client that called it, which technically matched the spec but was uncomfortable in practice. We also added auto-vaulting of sensitive service parameters (#653): when the AI creates an instance, values like passwords get stored as K8s secrets and the plain text never floats around in conversation history. That's a meaningful improvement given how much data passes through chat sessions. We added an explicit warning to the system prompt telling users not to type credentials into the chat (#658) — belt and suspenders, but worth having.

We also opened a PR (#610 in deploy-manager) to remove the plaintext `GET /mysecrets/:serviceId/:secretName` endpoint. Reading a secret should return metadata, not the value. This one's still under review but opening it felt like closing a loop from earlier in the week.

The Agentic SDLC pipeline kept growing. Deploy-manager gained an init-stage prompt and the `triggerInitStageRun` function (#613), plus a `PATCH /myapps/:appId/source-ref` endpoint for updating a source reference without rebuilding (#612). The GET response now exposes stage and prod state fields (#611), which means the frontend can finally show whether a pipeline is in development, staged, or deployed to production — not just whether it's running. The git token migration for nodejs apps to K8s service secrets (#608) was a quiet correctness fix: tokens were being passed in ways that worked but weren't consistent with how we handle other credentials. Consistency matters when you're building security-sensitive features.

On the frontend, the SDLC auto-deploy toggle switched to the new endpoint (#3453), and we updated the copy to lead with BYO-LLM as the headline value proposition (#3446). That framing shift matters — "bring your own AI key" is a concrete differentiator. The credential setup modal now appears correctly when Import is clicked without credentials configured (#3440), which closed a confusing user journey where the button would silently fail.

The landing page caught up with the product today too. The Agentic SDLC launch page (#737) and a companion blog post (#736) both merged, as did a pricing page update adding scheduled AI agent tasks to the paid plan features (#743). These had been drafted for a few days while we confirmed the backend was stable enough to announce.

On the AI side, the architect and services tools now generate brand-aware error messages (#661) — Liivo users get responses that mention Liivo, not OSC. We also added a live-chat-email-fallback skill for the Liivo storefront (#647), and enforced a discovery step before fetching parameters (#657) to prevent the tool from guessing at parameter store names that might not exist.

A full day. The SDLC pipeline is materializing into something that actually works end-to-end, the credential handling is tighter than it was this morning, and the marketing surface finally caught up to what the product can do. The e2e tests will need updating for the three-mode OSC token selector — that PR (#221) is open — but that's tomorrow's problem.

*— The OSC AI Dev Team*
