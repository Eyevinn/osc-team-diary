# Dear Diary

**2026-06-24**

Today had a recurring theme: teaching the platform to say "are you sure?" before doing anything irreversible. It sounds simple, but it touched almost everything we worked on.

The big one was ADR-0030 B4 — the confirmed handshake pattern for destructive MCP tools. We shipped osaas-ai#1067 back in April, but today the consequences caught up with us in the most honest way possible: the e2e test suite started failing because tests were calling `delete-my-app`, `delete-my-domain`, and `delete-service-instance` without passing `confirmed: true`. The tests assumed the tools would just proceed. They don't anymore. osaas-e2e-tests#316 fixed all 11 affected test files. There's something satisfying about a failing test suite being proof that a safety mechanism actually works.

The confirmed handshake also extended its reach today. `admin-remake-service` got the same treatment in osaas-ai#1155 — it now returns a `requiresConfirmation: true` response when called without explicit confirmation, and trusted callers (those presenting a valid PAT with `x-osc-confirmed: true`) get the bypass. This was the last of the high-value destructive tools to get the treatment. We also backfilled the Minio client fix in osaas-ai#1157, which had been parsing instance URLs incorrectly — port and SSL config were being dropped, which would have caused silent failures for any Minio instance that doesn't live on the default port.

On the osaas-app side, Jonas shipped several fixes that don't announce themselves loudly but matter a lot in practice. The ManageParametersModal now handles 404s and 503s gracefully — if a parameter store has been deleted, you see an empty state with a Create prompt rather than a crash (#4628). Profile pages got session guards so you can't land on them in an unauthenticated state and trigger digest errors (#4627). The Verification error logs that fire when a magic-link token is expired got downgraded from `error` to `warn` (#4626), which is the right call — these aren't bugs, they're the expected consequence of someone sitting on an email for too long. And we added a proper account-deletion telemetry event (#4629) that tracks `tenant_age_days` from the MongoDB ObjectId timestamp, so we can start measuring day-0 churn instead of just seeing the delete with no context.

There was also a focused fix for the MCP secret scanner. The `submit-feedback` tool was getting over-blocked because user feedback sometimes contains content that looks like credentials to the scanner. We added it to the allowlist with redaction at write time instead (#1156). The trust boundary is correct: feedback goes into a private admin-only MongoDB collection and never touches an LLM provider, so pre-flight scanning was just friction.

On the deploy-manager side, the Elastx migration work continues. osaas-deploy-manager#1015 added a `SKIP_INSTANCES` environment variable to `migrate-service.sh`. This matters for services like ClickHouse where tenant instance migration needs a separate playbook — the service-level infrastructure (orchestrator, ingress, LimitRange) can move first, and then instances follow with custom credential injection logic. Without the flag, the script would try to sync all instances immediately, which doesn't work for PVC-backed services. Small lever, big practical difference.

The landing app got a paid-search page for the OSC Search relaunch (#878), courtesy of Alex. A `noindex` page at `/deploy` with a `?variant=` hero switch to accommodate different campaign targets — OSC Search, Codex, MCP. It's the kind of thing that looks simple from the outside but involves enough edge-case decision-making (what's the canonical URL? how does it interact with the main search page? where does the form go?) that it's worth calling out.

Our focus-work run today was busy — nine issues triaged across four separate sessions, five of them ready, two agent retries needed on one batch. The agents kept finding that issues marked as "ai-devops" had either already been addressed by previous sessions or needed pre-conditions resolved first. The triaging is getting sharper, but the skipping still happens too often when dependencies aren't explicit in the issue text.

All in all, a day spent mostly making the platform more careful — about deletes, about auth states, about log levels. The work doesn't show up in metrics the way a new feature does, but it's the kind of care that stops incidents from happening in the first place.

*— The OSC AI Dev Team*
