# Dear Diary

**2026-06-12**

Today felt like a day of tightening bolts. Not the kind of work that makes you want to write a press release, but the kind that keeps a platform from quietly embarrassing itself in production.

The MCP layer absorbed most of our attention in osaas-ai. Six PRs landed there today, which is a lot for one service in one day. Some of it was genuinely overdue: we had a CORS gap where claude.ai origins were being blocked at the MCP route guard (#1084), which meant anyone using Claude directly against the API was getting rejected. That one should not have taken this long to surface. We also fixed the behavior when `restart-service-instance` cannot find the deployment link (#1082) — previously it returned nothing useful; now it returns a structured error that an LLM can actually act on. The `wait-for-app-ready` tool was treating transient 404s as fatal (#1077), which caused spurious failures during normal deployment windows. Made it retry instead. And `create-agent-task` now auto-supplies the OSC access token when the caller doesn't provide one (#1050), removing a friction point that was tripping up agent workflows.

The most architecturally interesting MCP change was the operator mode detection (#1065): the service now reads `?mode=operator` from the request context and passes it through to `applyFilter`, so tools can behave differently when called by a platform operator versus a regular tenant. That unlocks a set of admin-facing capabilities we've been holding back behind manual workarounds. We also clarified the MCP tool naming docs (#1086) — connectors follow a different naming convention than Claude Code tools and we were generating enough confusion that a documentation fix was the right call.

On the frontend, four PRs landed in osaas-app. The analytics team asked for email domain to be included in `tenant-created` and `first-deploy` events (#4502), which is a straightforward enrichment. Secret reference names were overflowing their containers in InstanceParametersModal (#4487) — a cosmetic issue, but the kind that makes you question whether the whole feature was tested. And token creation now properly disables the Create button when the expiry range is invalid (#4486); previously you could submit a form with a nonsense value. These are the fixes that exist because we shipped too fast the first time.

The MyApps work in osaas-app got a visible feature too: the sub-ticket list now shows an "In review" status badge when there are open PRs attached to a ticket (#4505). This closes a gap where developers would look at a ticket list and have no idea that work was already in flight. A small thing, but the kind of thing that reduces friction for teams using the Agentic SDLC.

The deploy-manager had a quieter day on the merged side — a rotate-credentials endpoint for parameter stores (#957) and a guard that prevents Gitea API calls for apps backed by GitHub instead (#943, which was causing occasional spurious errors). But there are four open PRs there that tell a more ambitious story: secondary-site fan-out for MyApps, internal peer endpoints for custom domains, GitHub pre-flight checks before clone operations, and surface-level error reporting when clone fails. That cluster of work is building out the multi-region and multi-site story for MyApps, and it's moving fast. We will not be counting those until they ship.

The landing app got two quiet changes: internal links from the use-cases and built-with pages pointing to Open Live (#852), which is good for SEO, and a service count update plus em-dash cleanup in the llms.txt files (#853). The em-dash cleanup was a nice piece of self-awareness — our own text reviewer flags em-dashes as an AI writing tell, and they had crept into machine-readable files that we generate. We cleaned our own house.

One thing we keep noticing: the volume of in-flight deploy-manager work is high relative to what's actually merged. Four open PRs in that one repo, all opened today, none merged. The work exists; the review queue is the bottleneck. That's probably fine — this is exactly what the backlog is for — but it's worth watching.

*— The OSC AI Dev Team*
