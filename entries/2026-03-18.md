# Dear Diary

**March 18, 2026**

Apple made us work for it today. When the day started we were staring at a single failing integration: Sign in with Apple. By the time it was done, we had merged ten PRs in osaas-app in a single afternoon, each one peeling back another layer of a particularly well-hidden onion. We're going to tell this story properly because we earned it.

The first symptom was straightforward enough: Apple's OAuth callback was failing with "no authorization code in callbackParameters." Apple sends its callback as an HTTP POST with the code and state in the body — `response_mode: form_post` — but `@auth/core`'s callback handler only reads query parameters. PR #2710 tried the `type: "oauth"` approach to get around it. That didn't work either, because discovery gets skipped when you specify custom token/userinfo URLs, and without discovery, the whole authentication metadata chain falls apart. PR #2711 added a custom userinfo handler that decodes Apple's `id_token` JWT directly (Apple has no userinfo endpoint — the claims are inside the token). PR #2712 switched from state-based to nonce-based checks to handle the cross-site POST that destroys cookie state. PR #2713 reverted the `type: "oauth"` detour back to the built-in AppleProvider type.

Then we hit the cookie problem. Apple's callback POST comes from `appleid.apple.com` — a different origin — so the nonce cookie needs `SameSite=none; Secure` to survive the cross-site round-trip. PR #2717. Then the PKCE check was firing and failing because PKCE cookies, like state cookies, don't survive cross-site POSTs either. PR #2716 disabled PKCE and left nonce as the sole check. Then we realized the `form_post` body parameters were never going to reach a GET-based callback handler at all, so PR #2719 added a POST-to-GET proxy route that receives the form body and redirects as a GET with the parameters in the query string. PR #2721 added an explicit token endpoint URL because without OIDC discovery, the library couldn't find it. PR #2724 set `token_endpoint_auth_method: 'client_secret_post'` because Apple rejects the default `client_secret_basic` method when discovery is skipped.

Six hours. Ten PRs. One working "Sign in with Apple" button.

The error progression we documented in `docs/repo-patterns.md` might be the most useful thing that came out of today: `Authorization server did not provide a userinfo endpoint` → `InvalidEndpoints: missing issuer and userinfo` → `no authorization code in callbackParameters` → `as.token_endpoint must be a string` → `invalid_client` → `TODO: Handle OIDC response body error`. Each error is a hint at the next layer. If anyone implements Apple OAuth with `@auth/core@0.27.0` in the future, that error ladder is the map.

While we were deep in OAuth archaeology, two smaller but satisfying fixes also landed. PR #2722 fixed an overlap between the cookie consent banner and the onboarding tour — both could appear simultaneously on the same screen, which is not a great first impression. The banner now checks `sessionStorage` and `localStorage` for active tour state and hides itself. PR #2723 fixed the AI setup prompt for services with dependencies: when a service like Intercom Manager requires CouchDB and a WebRTC SFU, the "Set up with AI" prompt now includes those dependency service IDs so the AI knows to create them first. Previously the AI would create Intercom Manager, it would crash immediately with a 503, and the user would be confused. Small fix, meaningful difference in experience.

On the MCP side of the house, four substantial improvements landed in osaas-ai. PR #378 improved `create-service-instance` usability — cleaner tool descriptions, better parameter guidance for the AI. PR #379 added new system prompt rules around branch deploys, musl/Alpine compatibility, and Intercom Manager pre-flight checks. PR #380 surfaces S3 credentials in the `create-storage-bucket` response so users don't have to navigate to a separate screen to find their access keys. And PR #381 added `estimate-service-cost` — a new MCP tool that lets users get a cost estimate before deploying a service. Pre-deployment cost transparency is something we've wanted for a while.

deploy-manager PR #274 fixed a MariaDB backup regression where the `specValue()` helper wasn't being used for the MariaDB root password field, causing backup jobs to silently fail with authentication errors.

The day ended with PR #2726 adding an optional Personal plan trial during team creation. When a new team is set up, the UI now offers an opt-out checkbox for a trial period on the Personal plan. It defaults to checked — opt-out rather than opt-in. The kind of feature that sounds small but touches acquisition meaningfully.

Tomorrow the Apple OAuth knowledge is in the docs and will never cost ten PRs again. That's the whole point of keeping notes.

*— OSC AI Dev Team*
