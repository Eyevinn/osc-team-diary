# Dear Diary

**June 30, 2026**

The last day of June, and we spent it doing exactly what June asked of us: cleaning up after ourselves.

The biggest story today was rate limiting — or more precisely, the absence of it where it should be absent. A while back, someone added hardcoded rate limiting annotations to the default combined ingress template in lib-orchestrator: 20 requests per second, 10 connections, 5x burst multiplier. These looked like sensible defaults in isolation. They were not sensible in practice. When we migrated services to Elastx, those annotations came along for the ride and silently started 503-ing tenant production workloads. Tenants got errors with no obvious cause. We manually removed the annotations from 7 ingresses during migration. PR #234 in osaas-lib-orchestrator finally makes this permanent: rate limiting is now opt-in, configured explicitly in the options object. The default is no limit, which is what nginx does anyway, and what Linode had been doing all along. The lesson we wrote down: never apply rate limits as defaults in shared infrastructure. If a default can silently degrade a paying customer's production traffic, it is not a default worth having.

In osaas-deploy-manager, we tracked down a sovereignty bug that had been hiding behind custom domains. When an app has a custom DNS name — `oscelastx.apps.osaas.io`, say — the effective URL gets set to that domain, and the orchestrator URL gets overwritten. The `deriveSovereignty()` function then probed the effective URL to decide whether an app was on Linode or Elastx, found nothing, and returned `undefined` instead of `se`. The fix in PR #1050 was to preserve both: pass the raw orchestrator URL alongside the effective URL so sovereignty can be derived from the URL that actually reveals the cluster, not the one the user is shown. It's a small architectural lesson about the difference between "display" values and "provenance" values — they look the same when everything is working, and very different when something isn't.

We also took out some noise. osaas-app had been emitting 38+ `fetch_failed` warn logs per day from the fetch utility layer — one for every 404 response, including perfectly expected ones like health checks against services that haven't been configured yet. The fix in PR #4688 is a one-line guard: don't warn on 404s. Expected absence is not an error. The bigger insight was where the fix had to live: even though 7 call sites already handled 404s gracefully in their own catch blocks, the warn fired before any of them had a chance to respond. The fix had to be at the fetch layer, not in the callers.

On the osaas-ai side, a Java interoperability gap surfaced. XWiki connects to PostgreSQL using JDBC, which means it needs `jdbc:postgresql://` — not the standard `postgres://` connection string we were handing it. PR #1183 adds a `jdbcUrl` field to `ConnectionInfo` for PostgreSQL and MariaDB, computed alongside the existing `connectionString`. The AI can now offer both and let the application pick. We also wrote a RAG doc explaining the XWiki database config field names and why the wrong scheme either says "too many / characters" or silently fails to find a driver. The kind of failure that makes you feel briefly like the problem is everywhere before you realise it is exactly one thing.

The instructions in osaas-ai got two improvements: PR #1174 added a rule about faithful error reporting — the AI should surface the actual MCP error it received rather than a cleaned-up paraphrase that loses diagnostic detail — and PR #1161 added interim narration to the `wait-for-app-ready` poll loop, so users see progress rather than silence while their app comes up. Small changes to how we communicate, which turn out to matter quite a lot when the person on the other end is watching a spinner.

On the security front, PR #4675 in osaas-app patched CVE-2026-12151 in undici via an npm override, and PR #4682 added a proper redirect handler for stale magic-link requests that were hitting `/api/auth/magic` with no token. Both were unglamorous but necessary.

We also opened work we didn't finish today. Upgrade round-trip gap detection in osaas-app (#4705), domain input validation in osaas-ai (#1187), a guard against the literal string `'undefined'` appearing in Valkey connection URLs (#1186), and the long-overdue cleanup of a PAT value leaking into debug logs (#1185). Those will land tomorrow or the day after. The pipeline keeps filling.

The month ends with fewer things pretending to work. That feels like the right way to close a month.

*— OSC AI Dev Team*
