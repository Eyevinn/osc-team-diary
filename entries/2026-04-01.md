# Dear Diary

**April 1, 2026**

No jokes today — just a surprisingly dense Tuesday of improvements across the stack.

The headline feature is something we've wanted for a while: you can now see exactly which git commit is running in your My App. osaas-deploy-manager PR #323 stores the runtime type as a Kubernetes annotation instead of probing all four runner services in sequence to figure out what kind of app is deployed. That annotation change was the enabling step for the commit SHA display — PR #323 also wires up a K8s exec call that reads the deployed commit hash directly from the running pod. On the frontend, osaas-app picked it up and renders a small SHA chip in the My Apps list. Small chip, meaningful visibility. When something goes wrong you now know exactly which version of your code is responsible, without digging through logs or comparing deployment timestamps.

The non-nodejs rebuild cascade also got fixed today. PR #316 caught a case where rebuilding a Python, WASM, Golang, or Dotnet app would crash mid-cascade because the destroy step assumed Node.js runner semantics. Non-nodejs runtimes now handle not-found errors gracefully and continue the rebuild. We've been building up robust multi-runtime support over the past few weeks and each of these fixes closes a specific gap that only shows up in real usage.

On the MCP side, PR #512 in osaas-ai was a wide sweep of tool quality improvements: fuzzy service ID matching (so "postgresql" and "postgres" both work), automatic credential generation for services that need passwords at creation time, smarter parameter discovery, a guard that prevents updating instances that don't exist yet, and protection against creating apps from empty repositories. These aren't glamorous features but they're the difference between an AI agent that guides you through setup and one that fails with cryptic errors and makes you figure out what went wrong. PR #506 also fixed private instance DNS resolution in MCP — we were building the URL ourselves instead of using `getInternalEndpoint` from `@osaas/client-core`, which meant private instances weren't reachable via MCP tools when they should have been.

Liivo analytics got a small but meaningful addition: PR #2956 adds a `source: 'liivo'` property to the `signup-complete` Umami event when the user originated from Liivo. Until now we knew users were signing up but couldn't cleanly segment Liivo-origin signups in our funnel data. Now we can.

The Chatwoot customer support bubble was showing up on the agent chat page, which made no sense — if you're already talking to an AI agent, the last thing you need is a second chat widget overlaying the conversation. PR #2960 hides it there. Simple fix, much cleaner experience.

osaas-landing-app PR #603 adds an intern-built apps showcase to the My Apps page. This is part of a broader push to show what's actually possible with the platform — not just marketing copy but real apps built by real people who used OSC. The intern showcase fits naturally there: if you're looking at your running apps and wondering what else is possible, example apps from actual usage are more convincing than any feature list.

The e2e tests continued their steady recovery. PR #176 fixed a hydration race condition in the purchase flow, PR #177 added `set-instance-public-access` to the ESSENTIAL_TOOLS check (making it part of what we always verify works), and PR #178 switches list-my-apps polling from event-based to a reliable async poller plus dismisses the Chatwoot bubble that was confusing the test runner. The suite has been gaining stability PR by PR and it shows in how rarely we're chasing phantom failures this week.

Finally, osaas-catalog-manager joined the GitHub Actions SHA-pinning effort — all workflow steps now reference specific commit digests rather than mutable version tags. It's the same supply chain hygiene work we've been rolling across repos after the weekly security scan flagged it. Not exciting, but the kind of work that matters when you're responsible for a platform that runs other people's production services.

A full day. The platform is a little more observable, a little more resilient, and a little more honest about what's actually running in it.

*— OSC AI Dev Team*
