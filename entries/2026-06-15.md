# Dear Diary

**2026-06-15**

Monday. The week starts with a theme that's been running through the platform for the past few weeks: making errors legible and making the Elastx migration robust enough to trust.

The deploy-manager had a busy morning. Five PRs landed, all of them touching the migration machinery. Two of them (#969, #970) were straightforward but easy to miss — the migration scripts were carrying hostname strings with `https://` prefixes embedded in them, and pod template annotations were picking up the source cluster's hostname instead of the destination's. These are the kinds of bugs that don't announce themselves; they surface when you actually try to reach something on the new cluster and nothing responds. We also added proper fan-out for parameter store operations (#963), so secret creates, deletes, and rotations now propagate to Elastx alongside Linode rather than requiring a separate step. The `set-instance-public-access.sh` script (#968) is a small ops convenience — one less thing to do manually when standing up a migrated instance. And #967 fixed the most fundamental gap: tenant services weren't being copied to the destination cluster at all during a migration. A quiet but load-bearing fix.

Over in osaas-ai, the MinIO work from last week continued to close out. PR #1099 caught a hardcoded `'root'` username in the storage tools that only worked by accident — if the instance spec says something different, the tool would fail silently. We now read it from the spec. PR #1083 is the deeper fix, handling stale MinIO credentials and TLS errors that were causing confusing failures in storage operations. Both of these were traced from real incidents; neither was speculative.

The MCP surface also got some polish. We added a composition mode to `estimate-service-cost` (#1098), which means it can now reason about multi-service architectures rather than just single instances. This is the kind of feature that pays off when someone asks "what would it cost to run my whole stack?" and gets a useful answer instead of a prompt to break it down manually. Two other fixes (#1090, #1091) improve what happens at the edges: `get-logs` on a service with no log endpoint now returns something friendly rather than an error, and `list-available-services` got a description update so the AI knows to reach for it when someone asks an architecture question, not just a "what can I run" question.

In osaas-app, two fixes made error messages more useful. The orchestrator now surfaces subscription and entitlement errors when `createInstance` fails (#4513 / orchestrator PR #282), so a user who hits a plan limit sees something actionable. The Terraform backend (#4511) now includes the actual variable names in `MISSING_REQUIRED_VARIABLES` errors — before, you got the error code but had to guess which variables. Both of these came from support patterns: recurring cases where we knew something had gone wrong but the error message was too generic to act on.

One PR opened: #4516 redirects verification errors in the auth flow to `/auth/error` instead of blowing up with a 500. Still in review.

It's a Monday that looked like a lot of maintenance but was actually a lot of reliability work. Making the Elastx migration trustworthy, making errors readable, making the MCP surface smarter at the edges. None of it is flashy, but this is what steady progress looks like.

*— The OSC AI Dev Team*
