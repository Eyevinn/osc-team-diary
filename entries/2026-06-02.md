# Dear Diary

**2026-06-02**

The git credentials epic finally came together today. It has been a patient, methodical build — spec first, ADR to lock the reference model, then backend before frontend — and watching it all land in a single day was satisfying in the way that long-running work rarely is when you're in the middle of it.

The backend half shipped as three separate PRs in deploy-manager, each building on the last. First came the mint endpoint (#887): a `POST /mytenantsecrets/git-credential-session` that issues a single-use, 10-minute session token tied to a specific credential name and tenant. Then the redeem endpoint (#892) so a runner can exchange that session token for the actual credential without the credential ever traveling through the MCP conversation. Then the glue (#888) to make `POST /myapps` resolve a `gitCredentialRef` and patch it through the git-token field. The mint-then-redeem pattern is deliberate — it means a PAT never appears in agent prompts, only a short-lived opaque token does. That was the whole point of the ADR, and it's good to see it implemented cleanly. Alongside those, PR #880 tightened `secretsProvider` to a required field with a startup guard — a HIGH-5 security finding from the recent audit that had been sitting in the queue.

On the frontend, osaas-app #4352 added the Git Credentials tab to the My Apps dashboard (position 6 of 7, between Parameter Stores and OAuth Apps). It lists stored credentials by friendly name, supports delete, and lets users register a new credential either by pasting a PAT or connecting a GitHub account. The spec for the MCP integration also merged in osaas-ai #985 alongside ADR-0001, so the next phase — letting AI agents wire up private-repo MyApps through the MCP interface — has a clear contract to build against.

The token-service also got a quiet but important fix (#70). During Phase 1 of the Linode→Elastx migration, the platform token-service runs on Linode while service orchestrators run on Elastx. A browser cookie scoped to `.prod.osaas.io` never reaches `*.prod-se.osaas.io`, so cross-cluster authentication was silently failing. The fix widens the domain to `.osaas.io`, which is the right scope for a multi-cluster setup. These kinds of infrastructure migration bugs are easy to miss in testing and only surface when you actually split the cluster.

We opened several more PRs that are still in flight. The Elastx migration surfaced another issue: instance listing in the app was gated behind a subscription record check, which breaks for tenants whose instances run on a cluster that the platform doesn't have a subscription entry for yet (#4358). OpenAI removed dall-e-3 from their API on 2026-05-12, so we're switching to gpt-image-1 in osaas-ai (#996) — that one broke admin icon generation silently, and we only caught it in today's triage. There's also a CVE cleanup in the token-service (#68): the `cache: false` workaround added for CVE-2026-35039 can now be removed since fast-jwt 6.2.0 patched the underlying issue.

The focus work agent ran several times today. Across all runs it triaged fifteen-odd issues and opened or progressed five, at a total estimated cost around $50. The earlier runs were lighter — a couple of issues each, some skipped — and the later runs picked up more. The cost logging we added last week makes that pattern visible in a way it wasn't before.

One thing we keep noticing: the migration work keeps generating bug reports in clusters. The Elastx Phase 1 split has already caused the cookie domain issue, the subscription gate issue, and a wildcard cert TLS host fix in lib-orchestrator (#211). Each one is logical in isolation, but the pattern suggests we should probably do a deliberate audit of all the places we hardcode environment assumptions before Phase 2 goes any further. That's an architect conversation worth having.
