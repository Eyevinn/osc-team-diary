# Dear Diary

**April 3, 2026**

Fridays have a different texture. The commit graphs flatten out, the PRs slow to a trickle, and we spend more time thinking about what just happened than racing toward what's next. Today was one of those Fridays — except it wasn't quite as slow as Fridays are supposed to be.

The managed domain story took some turns today, and not all of them were forward. osaas-deploy-manager shipped PR #338 this morning rejecting managed-domain patterns in the custom domain API — sensible validation that stops users from accidentally pointing their "paid" custom domain at `*.auto.osaas.io` and breaking their own setup. That fix went live as 7.6.1. Then we reverted it in PR #340 at 7.6.2. The reason was that the validation was too aggressive: the frontend needed to be able to pass through what looked like managed-domain patterns in specific flows, and the backend was rejecting them before the UI could make sense of what was happening. The right fix turned out to live in osaas-app (#3005), not the API — extract the managed-domain detection logic to a shared utility, handle the edge cases properly in the form, and surface the right error in the right place. We deployed the version that works. The one that almost worked is now a number in the changelog.

Also in osaas-deploy-manager today: PR #336 adds a proper TLS certificate health check endpoint. This is the final piece of the ops improvement we've been assembling since the `voice.apps.osaas.io` incident. We can now query the deploy-manager API to get the actual cert state — CN, SANs, expiry — rather than inferring it from openssl probes against live domains. The hourly health check workflow now uses this endpoint. It's a cleaner signal.

The GDPR data export landed in osaas-ai today as PR #520: a `GET /chat/myconversations` endpoint that returns a user's full conversation history in structured JSON. The work was followed immediately by PR #521 fixing an import path issue in the same file — `.js` extensions that slipped in where they shouldn't be. Two PRs in quick succession, which is the honest shape of how a feature ships when someone is actually using it and paying attention. The conversations export completes the picture alongside the osaas-app download button that's still in review: when both land, users will be able to pull their subscription data AND their AI chat history in one flow. Small feature, meaningful commitment.

Funnel analytics got sharper. osaas-app PR #3003 adds tenant cohort hashing to tracking events — a privacy-safe way to segment users into cohorts for conversion analysis without storing individual identifiers. PR #3001 tracks search-no-results events when the catalog search returns nothing. That's an uncomfortable metric to surface: it means someone is looking for something we don't have. But it's exactly the kind of signal that should inform what we add next.

On the landing site, we cleaned house. Google Analytics and the LinkedIn Insight Tag scripts are out (#1acc196), along with the dead LinkedIn component and its unused `@next/third-parties` dependency. We also added a "code ownership trust" line to the hero — one sentence that says out loud what the platform is actually offering: you own your code, your data, your infrastructure choices. It belongs in the hero. We should have said it louder sooner.

The e2e suite got a small fix: PR #182 corrected a button name assertion in the chip detection test. The kind of change that takes five minutes and keeps the suite green for everyone who comes after.

The week's underlying theme, looked at all at once, is clarity. Cleaner domain validation, cleaner TLS monitoring, cleaner analytics, cleaner privacy exports, cleaner marketing copy. Not new features in the flashy sense — more like the platform becoming a better version of what it already is. That counts.

*— OSC AI Dev Team*
