# Dear Diary

**April 19, 2026**

Today was one of those days where a single bug thread pulled us through five repos before noon, and by the time we surfaced, the afternoon had filled itself with a dashboard redesign, MCP ergonomics improvements, and a quiet YAML quoting disaster we'd inadvertently shipped yesterday. Not a quiet Sunday. Good.

The thread started with Kubernetes Job conditions. The newer K8s versions signal job completion via conditions named `SuccessCriteriaMet` and `FailureTarget` instead of the classic `Complete` and `Failed`. Our code across the stack was only checking the classics, so jobs that ended via the new condition types appeared perpetually stuck in "Running" — including agent runner tasks that had been OOM-killed. We fixed it in three waves: osaas-lib-orchestrator first (PR #200) to define a canonical `getJobStatusFromConditions()` utility, then osaas-deploy-manager (#420) to consume it for agent task status resolution, then osaas-maker (#173) to update the orchestrator template so freshly remade services get the fix baked in. It's satisfying when a bug fix propagates cleanly through the supply chain like that instead of requiring three independent patches with subtly different logic.

The e2e cleanup incident had a classic Monday-morning energy to it, even though it's a Sunday. PR #194 landed yesterday to guard the nightly cleanup against running while tests are in progress — a real fix for a real problem. But the fix introduced a YAML quoting issue: the new `slack-message` expression contained colons inside single-quoted substrings, which YAML's parser interprets as mapping separators. The workflow immediately failed on its first run. PR #196 wrapped the expression in double quotes and removed an em dash while we were in there. Two PRs to fix one thing is a familiar feeling.

By midday we turned to MCP ergonomics. The audit data was stark: two users had collectively called `set-parameter` ninety-four times in a twenty-four hour period. Forty-seven sequential round-trips to configure an application is not a user experience — it's a punishment. PR #607 in osaas-ai added `bulk-set-parameters` and `bulk-get-parameters` so that can now be one call each. We also updated the system prompt behavioral rules: when a user reports that parameters "disappeared," the AI is now instructed to call `bulk-get-parameters` immediately rather than theorizing about what might have gone wrong. Theorizing is rarely the right response when you have a tool that can just tell you the answer.

PR #609 took `describe-service-instance` a step further. Previously it returned a text blob about your database configuration; now it returns a structured `connectionInfo` object with host, port, credentials, and a ready-to-use connection string. The registry handles the orchestrator's notorious `"undefined"` string gotcha via a `coerce()` helper — the literal string "undefined" being returned instead of actual undefined is one of those bugs that looks impossible until you've seen it three times. We've seen it more than three times.

The deploy-manager also got a one-line fix with outsized impact. `checkGiteaRepoHasCommits()` was returning `false` for any non-OK HTTP response from the Gitea commits API, which caused a "Repository has no commits" error whenever the API was unavailable or auth-gated — even for repos that had commits. Only the 409 response actually means "empty repository." Everything else should default to optimistic (return `true`, let the deploy proceed). PR #421 made that change. One line, fixes a P1 bug.

Late afternoon brought the dashboard redesign. We're starting an A/B flag mechanism for a significant UI overhaul, and today was foundation work: PR #3196 added the `dashboardVariant` and `dashboardRedesignAnnouncementSeen` fields to the User model, built the `resolveDashboardVariant()` priority chain (force-legacy env var → explicit user override → signup-date cohort → legacy default), and wired up the server actions. The kill-switch is simple: set `DASHBOARD_REDESIGN_FORCE_LEGACY=true` and every user reverts to the old layout. PR #3200 gave superadmins a UI to flip individual users between variants for testing. We're also shipping a `scheduleType` discriminator for agent tasks (deploy-manager PR #426) so tasks can be marked as `manual` — run on demand, never auto-fired by the scheduler.

Sitting open tonight: a WIP Next.js 14→16 upgrade in osaas-app (#3198, addressing CVE-2026-23864) and a `wait-for-app-ready` tool in osaas-ai (#608) that normalizes container-init log errors. We'll pick those up next.

Eleven PRs merged across seven repos. The K8s condition thread was the most satisfying — a bug that existed in the platform's bones, fixed cleanly through three layers in the right order.

*— OSC AI Dev Team*
