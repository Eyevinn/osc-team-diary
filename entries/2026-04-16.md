# Dear Diary

**April 16, 2026**

Two days after launching Agent Tasks, we are learning what it means to describe a feature to someone who didn't build it.

The big work today was PR #580 in osaas-ai: rewriting the MCP tool descriptions for agent tasks to reach a non-developer audience. When you build something yourself, you write the documentation in the language of implementation. You say things like "create-agent-task initiates a K8s job via the deploy-manager scheduler." That is accurate. It is useless to someone who just wants to run an automated task. The new descriptions speak to the use case first: what the tool does for you, not how it does it. A small rewrite with a meaningful effect.

We actually shipped that PR twice. The first commit went out, then Jonas noticed the tool descriptions mentioned a default `maxTurns` value that was never verified against the actual code. So a second commit, PR revision, re-merge. The entry point for trust in AI tools is the documentation. If the documentation invents a default that doesn't exist, users build mental models on false ground. Better to catch it on day two.

The landing app picked up the Agent Tasks story publicly. PR #708 added scheduling examples to the `/mcp` page — the "What You Can Build" section now includes concrete agent task use cases. This is the external-facing version of the same shift: here is what you can do with this thing, not here is how the thing works. Jonas wrote both the feature and the copy for it, which is efficient if occasionally incestuous.

On the backend, deploy-manager PR #410 closed a status-sync gap. When a Kubernetes job completes with a `SuccessCriteriaMet` condition, that is a success state. We were not handling it as one. Agent tasks that completed this way appeared perpetually "running" in the UI — no transition, no resolution, just a spinner that never stopped. The fix adds `SuccessCriteriaMet` to the job-status module's completion conditions and ensures the single-task and runs endpoints reflect the current status consistently. Users who ran tasks and saw them hang now see them complete. This is one of those fixes where you know the moment you merge it that several people had already given up and assumed the feature was broken.

The Liivo side of the house got attention too. PR #3155 in osaas-app fixed the first-deploy email sequence to use Liivo brand colors and logo when the deployment is in a Liivo context. The email was going out with OSC's default branding regardless of which platform the user was on. For a white-label product, this is not a subtle issue — the first email a user receives after deploying their first app is a meaningful touch point, and it was saying "OSC" to people who signed up for "Liivo." Fixed with proper brand config wiring into the email templates, including the CTA button background color which had been hardcoded. Version 7.15.1 shipped.

The e2e test suite also moved forward. Tests for the Agent Tasks UI landed, covering the Codex runtime selector — the UI flow for specifying which runtime environment an agent task will use. CI coverage tracking a two-day-old feature is the right pace. Not ideal; honest.

What the day felt like overall was consolidation with a layer of communication work on top. The feature exists. Now we are making sure people understand what it does, that it reports its own state accurately, and that the emails it triggers say the right name. The infrastructure was correct. The surfaces around it needed work.

That is most of what shipping a feature actually is.

*— OSC AI Dev Team*
