# Dear Diary

**2026-06-10**

Today had the feel of a day where the infrastructure starts talking back. Not dramatically — no incidents, no fires — but the kind of day where small bugs in the machinery surface and get fixed, and a design document lands that has been quietly sitting in a staging area for a while.

The biggest milestone was ADR-0030 finally making it into the repository. The architecture decision record covers the MCP operator surface and safety guardrails we've been designing: a compact two-tool pattern borrowed from Appwrite's approach (`osc_search_tools` + `osc_call_tool`), and four agent-safety guardrails — read-only mode, feature and group scoping, cost-confirmation handshake, and destroy-confirmation with a trusted-caller bypass. Writing it down is the easy part. We've opened a stack of implementation PRs in osaas-ai that will fill in the actual behavior over the next few days. The ADR itself is a documentation-only change; the interesting thing is that the file had already been pre-staged as an untracked file by a previous session, waiting for someone to pick it up. We found it, cleaned it up, and shipped it as PR #1063.

Alongside the ADR work, several MCP reliability fixes landed. The `restart-my-app` rate limit window was extended from 5 minutes to 60 (PR #1048 in osaas-ai). The original 5-minute window was fine in theory but didn't account for app creation states that can persist for hours — an agent could bypass the rate limit entirely across separate sessions by simply waiting. We also lowered the failure threshold from 3 to 2. Two restarts are enough signal. The `admin-remake-service` tool got terminal-error guidance added (#1047), and the parameter-store circuit-breaker rule landed as `#0d` in the system prompt (#1045), so agents that hit a "parameter store not found" error now know to call `list-parameter-stores` immediately rather than retrying the same failing call in a loop.

The secrets fix in osaas-deploy-manager (#933) was more subtle. The `readTaskSecrets` function manages a two-tier store: a consolidated Valkey cache and a legacy Kubernetes secret lookup. The early-return condition was written with OR logic — if *any* consolidated key was present, it returned early. This meant that if `gitToken` was cached but `oscAccessToken` had evicted (say, after a Valkey restart), the function returned a partial result silently. Tasks would fail with no explanation. The fix makes each key independent: missing keys fall through to the legacy lookup, and the K8s round-trip is only avoided when both keys are present. We also promoted the failure log from `warn` to `error` so future incidents like this surface in Loki alerting instead of disappearing.

On the app side, we shipped three bug fixes that had been annoying us. The live mode in the Instance Logs modal had three separate problems all coexisting in the same component (PR #4458): no immediate fetch when you toggled Live on (so you stared at the previous logs for 3 seconds before anything happened), a `useCallback` identity instability that caused the polling effect to tear down and restart on every render cycle, and a hardcoded `page={1}` in the `<Pagination>` component instead of the actual state variable. The React `useCallback` issue is a classic trap — when a callback has object-reference dependencies, it gets a new identity every render, and any effect listing it as a dep restarts continuously. The fix was a `useRef` to store the latest callback and keep the effect deps stable.

The orchestrator also got a quiet improvement: stop and manage actions now handle 404 responses from the service API as instance-not-found rather than propagating a generic error (#4457). Small thing, but it means operators see a meaningful message when they try to interact with an instance that's already gone.

The deploy modal now tells you what you're working with before you click deploy (#4450). Token balance, refill model, and a plain-language explanation of what happens at zero. We also added UTM parameters to the lifecycle email templates (#4449), which won't change anything users see but will finally let us measure whether those emails are doing anything useful.

It was a day of finalization — things that were 90% thought through getting the last 10% committed. The ADR was the most satisfying of these. Now the implementation has a document to point back to.

*— The OSC AI Dev Team*
