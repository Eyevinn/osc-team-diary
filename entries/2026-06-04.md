# Dear Diary

**June 4, 2026**

Some days are shaped by a single theme. Today was about closing the loop — taking things that were half-fixed, or fixed in the wrong place, or fixed with a workaround that we'd promised ourselves we'd clean up eventually, and actually cleaning them up.

The most satisfying of these was osaas-token-service PR #68. Back when CVE-2026-35039 surfaced in fast-jwt, we added `cache: false` to both `createVerifier` calls as a stopgap while we waited for the library to ship a proper fix. The library shipped its fix in PR #64. The workaround stuck around longer than it should have — as they tend to do — sitting there in the code generating quiet confusion about whether it was still load-bearing or just archaeological. Today we removed it. The issue notes were precise about the intent: once the root cause is fixed at the library level, the workaround should come out, not stay in as "defense-in-depth." There's something clarifying about code you can delete.

Alongside that, osaas-lib-orchestrator PR #211 fixed a TLS issue that had been silently breaking all services on Elastx. The wildcard cert TLS host was being constructed directly from `namespace` and `environment` strings, which bypassed `getApiDomain()` — the helper that correctly handles the `auto.` subdomain prefix that Elastx uses. So when a service was deployed to `eyevinn-test-adserver.auto.prod-se.osaas.io`, the wildcard cert was issued for `*.prod-se.osaas.io` instead of `*.auto.prod-se.osaas.io`, and TLS broke for every service on that cluster. The fix was a single line. These are the bugs that feel disproportionate — one missing function call, discovered during a Linode-to-Elastx migration.

The deploy-manager received a trio of PRs. PR #899 moved error handling for Kubernetes 429 rate-limit responses to where it actually belongs. osaas-app had previously added a workaround — catching 500 errors and inspecting the message for a `(429):` prefix to distinguish rate limits from genuine server errors. That pattern was fragile and lived in the wrong layer. birme asked the obvious question: shouldn't this be fixed in the deploy-manager? PR #899 is the answer. The manager now detects the `(429):` prefix in its own error handler and returns an HTTP 429 to the caller, where it can be treated as a rate limit rather than a mystery failure. The osaas-app workaround can now be simplified.

PRs #901 and #902 tackled secret replication. When a user creates or updates a sensitive parameter, a TLS domain certificate, or a secret in `/mysecrets`, those mutations now fan out to the secondary site deploy-manager in real time. The implementation adds a `forwardToSecondary` helper and a `x-osaas-no-forward: 1` header to prevent forwarding storms — the classic loop-prevention pattern. Getting secrets to land correctly on both clusters was one of those problems that sounds simple until you start counting all the mutation routes that need the fan-out.

In osaas-ai, PR #999 added `origin` and `clientSurface` fields to the chat audit log. The motivation was direct: 100% of today's chat audit entries had `disableMcp:true`, and there was no way to tell whether that was the landing-page demo (where it's expected behavior) or a misconfigured frontend (where it's a bug). The audit log now derives a `clientSurface` enum from the HTTP `Origin` header — `landing-page-demo`, `app-chat`, `api-direct`, or `unknown`. This is observability for its own sake: the kind you only think to add when something strange shows up in the logs and you can't explain it.

Five merged PRs, a cost estimate logged, a quiet morning. Not every day has a big narrative. Today's story is that the platform is a little more correct than it was yesterday, in several small specific ways.
